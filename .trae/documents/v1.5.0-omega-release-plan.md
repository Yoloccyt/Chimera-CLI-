# Chimera CLI v1.5.0-omega 正式发布执行计划

## Context

用户要求使用 Superpowers 工作流完成 v1.5.0-omega 正式发布流程,包括质量保障测试、发布说明审查、版本号同步、构建产物生成、安全扫描、变更日志、发布前检查审批、分发渠道发布,以及终端安装指南。

当前仓库已处于发布准备的最后阶段,但存在一个关键阻塞:本地 tag 与远端 tag 指向不同 commit,需要先由用户决策处理方案。

**关键发现**: 本地 master 上有 4 个未推送的 CI Windows GNU 修复 commit,且远端已存在 `v1.5.0-omega` tag 指向旧 commit。直接推送本地 tag 会被拒绝,必须先解决 tag 冲突。

## Decision: 升级为 v1.5.1-omega

用户已选择 **方案 A: 升级为 v1.5.1-omega**。

原因:
1. 远端 tag 已存在,任何对 v1.5.0-omega 的覆盖都可能造成协作方混乱。
2. 4 个 CI 修复 commit 是真实的发布准备变更,值得一个补丁版本号。
3. 避免 force 操作,符合项目 Git 协作规范。

本计划按 v1.5.1-omega 执行。

## Execution Steps

### Phase 1: 预检 (Step 0)

```powershell
# 确认工作区干净
git status --short

# 确认 tag 位置
git log --oneline -5
git rev-parse v1.5.0-omega
git ls-remote --tags origin 'v1.5*'
```

### Phase 2: 质量保证测试 (Step 1)

```powershell
# 设置环境
$env:RUST_MIN_STACK = '33554432'
$env:CARGO_INCREMENTAL = '0'
$env:CARGO_HOME = 'D:\Chimera CLI\.toolchain\cargo'
$env:RUSTUP_HOME = 'D:\Chimera CLI\.toolchain\rustup'
$env:PATH = "D:\Chimera CLI\.toolchain\cargo\bin;D:\msys64\mingw64\bin;$env:PATH"

# 1. 类型检查
cargo check --workspace

# 2. 单元/集成测试(Windows OOM 缓解 --jobs 1)
cargo test --workspace --jobs 1

# 3. 压力测试
cargo test -- --ignored --nocapture

# 4. Clippy 零警告
cargo clippy --workspace --all-targets --jobs 2 -- -D warnings

# 5. 格式化检查
cargo fmt --all -- --check

# 6. 本地 Release 构建与 UX 验证
cargo build --workspace --release
.\target\release\aether.exe --version
.\target\release\aether.exe --help
```

**失败处理**: 任何一步失败即停止发布,修复后重新从 Phase 1 开始。

### Phase 4: 构建产物与安全扫描 (Step 4/5)

```powershell
# 本地 Release binary
cargo build --release -p chimera-cli
.\target\release\aether.exe --version
(Get-Item .\target\release\aether.exe).Length / 1MB  # 应 < 50MB

# Docker 镜像
docker build -t chimera-cli:rc .
docker image inspect chimera-cli:rc --format '{{.Size}}'  # 应 < 104857600
docker run --rm chimera-cli:rc --version

# 安全扫描
cargo audit --deny warnings `
  --ignore RUSTSEC-2026-0190 `
  --ignore RUSTSEC-2026-0002 `
  --ignore RUSTSEC-2024-0436

# fuzz 配置静态检查
# 注意:Windows GNU 工具链无法编译 libfuzzer-sys,此步骤在 Windows GNU 上应跳过,
# 实际 fuzz 由 .github/workflows/fuzz.yml 在 Linux CI 执行。
cargo check --manifest-path fuzz/Cargo.toml
```

### 本地环境已知限制与缓解

| 限制 | 影响范围 | 缓解方案 |
|------|----------|----------|
| `cargo audit` 联网 fetch 失败 | 本地安全扫描 | 使用 `cargo audit --no-fetch ...` 读取本地 advisory-db；CI 仍在线执行 |
| 本地未安装 Docker | Phase 4 Docker 验证 | 跳过本地 Docker 步骤，`release.yml` 在 Linux runner 上构建并推送镜像 |
| Windows GNU 不兼容 libFuzzer | 本地 fuzz 编译/运行 | Windows GNU 跳过此步；实际 fuzz 委托 `fuzz.yml` Linux CI 执行 |
| stress_test 首/末单次比较易抖动 | Phase 2 压测 | 已修复为前后 100 次窗口平均 + 低基线绝对容差，见 `tests/e2e/stress_test.rs` |

### Phase 6: 分发发布 (Step 8)

```powershell
# 推送 master
git push origin master

# 删除旧本地 tag,打新 tag
git tag -d v1.5.0-omega
git tag -a v1.5.1-omega -m "Release v1.5.1-omega"

# 推送新 tag(触发 CI)
git push origin v1.5.1-omega
```

CI 将自动执行:
- `release.yml`: 5 平台 build + test + docker + GitHub Release
- `fuzz.yml`: 6 target × 300s fuzz
