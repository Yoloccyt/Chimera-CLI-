## 4. 依赖方向铁律审计

### 4.1 审计方法

读取 5 个代表性 crate 的 `Cargo.toml`,验证 §2.2 依赖铁律:
- `L(N) → L(N-1)` 向下依赖允许 ✓
- `L(N) → L(N+1)` 向上依赖禁止 ✗
- 跨层通信走 event-bus / mcp-mesh 允许 ✓

### 4.2 抽查结果

| 抽查项 | crate 层级 | 依赖方向 | 合规性 | 说明 |
|--------|-----------|---------|--------|------|
| nexus-core → 上层 crate | L1 | 仅外部 workspace 依赖(ndarray/serde/tokio 等),无任何 L2+ crate 依赖 | ✓ 合规 | L1 保持最小依赖,未向上引用 |
| chimera-cli → L1 | L10 → L1 | `nexus-core = { path = "../nexus-core" }` | ✓ 合规 | L10→L1 向下依赖允许 |
| event-bus 跨层引用 | L1 | 被 seccore(L4)/quest-engine(L9)/parliament(L8) 等多层引用 | ✓ 合规 | event-bus 是跨层通信唯一通道,被依赖而非依赖 |
| seccore → L5+ | L4 | 仅 `event-bus`(L1),无 L5+ crate 依赖 | ✓ 合规 | L4→L1 向下依赖允许,未向上引用 |
| quest-engine → L8 | L9 → L8 | `decb-governor = { path = "../decb-governor" }`(L8) | ✓ 合规 | L9→L8 向下依赖允许 |

### 4.3 补充验证: parliament (L8) dev-dependencies

parliament(L8)的 `[dev-dependencies]` 包含 `quest-engine`(L9),形式上为 L8→L9 向上依赖。但根据 §2.2 依赖铁律脚注:**"dev-dependencies 可绕过生产依赖方向(测试代码非生产代码),但仅限 `tests/` 目录"**。此依赖仅用于测试(`#[cfg(test)]`),不污染生产依赖图,合规。

### 5.2 4 类 Critical 事件 mpsc 旁路通道验证

**验证位置**: `crates/event-bus/src/bus.rs` 第 36-43 行

```rust
// bus.rs:25-44
/// 判断事件是否走 mpsc 旁路通道(Critical 安全告警事件)
///
/// §6.2 红线要求:Critical 安全事件(SkepticVeto/RedTeamAudit/AsaIntervention/
/// BudgetExceeded)必须用 mpsc channel 确保送达,避免 broadcast 在 Lagged 场景下
/// 丢失。
fn is_critical_mpsc_event(event: &NexusEvent) -> bool {
    matches!(
        event,
        NexusEvent::SkepticVeto { .. }
            | NexusEvent::RedTeamAudit { .. }
            | NexusEvent::AsaIntervention { .. }
            | NexusEvent::BudgetExceeded { .. }
    )
}
```

**结果**: 4 类 Critical 安全告警事件全部走 mpsc 旁路通道 ✓

| 事件 | severity() | is_critical_mpsc_event | 双通道保障 |
|------|-----------|----------------------|-----------|
| SkepticVeto | Critical | true | broadcast + mpsc ✓ |
| RedTeamAudit | Critical | true | broadcast + mpsc ✓ |
| AsaIntervention | Normal* | true | broadcast + mpsc ✓ |
| BudgetExceeded | Critical | true | broadcast + mpsc ✓ |

> *AsaIntervention 的 `severity()` 返回 Normal(因 severity() 是同步函数,不依赖运行时值),但其 Block 级别在语义上等价于 Critical,因此通过 `is_critical_mpsc_event` 强制走 mpsc 旁路确保投递。详见 `bus.rs:34-35` 注释。

### 5.3 双通道实现确认

**实现位置**: `crates/event-bus/src/backpressure.rs` 第 13-16 行

```
4 类 Critical 安全告警事件(SkepticVeto/RedTeamAudit/AsaIntervention/
BudgetExceeded)在 EventBus::publish/publish_blocking 中自动额外
投递到 mpsc 旁路通道。订阅者通过 subscribe_critical_events() 获取
mpsc Receiver,确保在 broadcast Lagged 场景下仍能接收。
```

**结果**: 失败 — `clippy::security` 不是标准 clippy 的 lint group

```
error[E0602]: unknown lint: `clippy::security`
  |
  = help: did you mean: `clippy::exit`
  = note: requested on the command line with `-W clippy::security`
```

### 6.2 处理

按任务指示"如果 clippy 不识别 -W security,记录并跳过此步",此项标记为 **SKIP**。

**补充说明**: 标准 clippy 的默认 lint 集已覆盖部分安全相关模式(如 `clippy::integer_arithmetic`、`clippy::panic`、`clippy::unwrap_used` 等),这些 lint 在项目日常 `cargo clippy --workspace --all-targets -- -D warnings` 中已强制为 deny(发布前检查清单第 2 项)。完整 clippy 检查由发布前检查清单覆盖,不在此报告重复。

### 8.2 遗留风险

1. **advisory-db 缓存可能滞后**: 本次扫描使用本地缓存(--no-fetch),advisory-db 最后更新时间未确认。CI 端(audit.yml ubuntu-latest)每日 UTC 02:00 自动 fetch 最新版本,可作为补充保障。建议在 tag 推送前确认 CI 端 audit job 最近一次运行通过。

2. **3 个 ignored RUSTSEC 警告**: 均经评估不影响项目(anyhow downcast_mut 未使用 / lru IterMut 未使用 / paste 仅宏拼接),但需持续跟踪上游修复进度。待 anyhow/ratatui 发布新版本后及时移除对应 --ignore。

3. **clippy::security lint group 不存在**: 标准 clippy 无此 lint group,安全相关 lint 依赖默认 lint 集(integer_arithmetic/panic/unwrap_used 等)在发布前检查清单中覆盖。

### 8.3 建议措施

1. **发布前**: 确认 `.github/workflows/audit.yml` 最近一次定时运行(每日 UTC 02:00)在 CI 端通过(本地 advisory-db 缓存可能滞后于 CI 端)。
2. **持续跟踪**: 定期检查 anyhow/lru/paste 上游是否发布修复版本,及时移除 `--ignore`。
3. **版本同步**: 确认 `Cargo.toml` workspace.package.version = 1.5.6-omega,CHANGELOG.md 存在 `v1.5.6-omega` 汇总章节(由代理 1B 负责)。
