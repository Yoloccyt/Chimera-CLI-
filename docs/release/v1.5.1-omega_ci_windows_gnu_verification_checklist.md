# v1.5.1-omega CI Windows GNU target 验证清单

> 本清单用于在首次推送 `v*.*.*-omega` tag 触发 `release.yml` 时,系统性验证 Windows GNU target(`x86_64-pc-windows-gnu`)的 MinGW 工具链配置是否在 GitHub-hosted `windows-latest` runner 上真正生效。
>
> 对应规则:`.trae/rules/nuxus规则.md` §10.5 P1 短板 "release.yml Windows job 未安装 MinGW"。
>
> 被验证配置权威源:`.github/workflows/release.yml` **L129-147**(MinGW 安装 + linker 环境变量覆盖 + PATH 注入),辅助验证步骤 L149-156(工具链软验证)、L183-210(构建)、L234-258(产物校验)。
>
> 本文件不修改 `release.yml`,仅作为验证操作手册与状态跟踪依据。常见错误的深度根因分析可进一步参考姐妹文档 [`v1.5.1-omega_ci_windows_gnu_verification.md`](./v1.5.1-omega_ci_windows_gnu_verification.md) §4.2-4.8。

---

## 1. 验证目标

**一句话**:确认 `release.yml` 的 Windows job 在 `windows-latest` runner 上,通过 `msys2/setup-msys2@v2` 安装的 MinGW 能成功完成 release 构建并产出符合契约的 binary。

**背景**:
- `release.yml` L129-147 已配置 MSYS2 安装 + `CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER` 环境变量覆盖 + `GITHUB_PATH` 注入
- 该配置逻辑上正确,但因 `build` job 仅由 tag 触发(L47 `if: startsWith(github.ref, 'refs/tags/v')`),从未在真实 CI 环境中跑通
- §10.5 标记为 `🔧 待下次 tag 推送验证`,本清单即为该验证的操作依据

**机制要点**:
- 本地 `.cargo/config.toml` 硬编码 `D:/msys64/mingw64/bin/gcc.exe`(本地 D 盘工具链)
- CI 上 `msys2/setup-msys2@v2` 默认安装到 `C:/msys64`
- 通过 `CARGO_TARGET_<TRIPLE>_LINKER` 环境变量(优先级高于 `.cargo/config.toml`)覆盖 linker 路径,无需修改仓库文件
- `GITHUB_PATH` 注入 `C:/msys64/mingw64/bin` 是为了让 cargo 能找到 `ar`、`windres` 等 MinGW 辅助工具(仅设置 linker 不足)

## 3. 验证指标

进入 GitHub repo → Actions → 对应的 "Release" workflow run → 点击 `Build (x86_64-pc-windows-gnu)` job,按以下顺序逐项核验。

每项均需明确记录:**通过 / 失败 / 跳过**,并附日志关键行截图或文本片段。

### 指标 a:`msys2/setup-msys2@v2` action 成功安装 gcc

| 字段 | 内容 |
|------|------|
| **检查项** | 步骤 `Setup MSYS2 and MinGW (Windows GNU target)`(release.yml L129-135)是否成功执行 |
| **预期结果** | 步骤退出码 0;日志中出现 `MSYS2 installed` / `Installing package gcc` 且无 error |
| **失败处理** | 见 §4.1。常见原因:网络瞬时失败重试即可;action upstream 回归则锁定到具体 commit SHA;MSYS2 镜像问题可加 `update: false` 跳过首次更新 |

**备注**:首次运行下载 MSYS2 完整镜像(约 200MB),耗时 3-5 分钟属正常。

### 指标 b:`CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER` 环境变量正确设置

| 字段 | 内容 |
|------|------|
| **检查项** | 步骤 `Configure MinGW linker for CI`(release.yml L137-147)是否成功写入 `$GITHUB_ENV` 与 `$GITHUB_PATH` |
| **预期结果** | 步骤退出码 0;`$GITHUB_ENV` 写入 `CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER=C:/msys64/mingw64/bin/gcc.exe`;`$GITHUB_PATH` 写入 `C:/msys64/mingw64/bin` |
| **失败处理** | 见 §4.2。本步骤日志较短,主要看退出码;真正生效需看指标 c/d 的 `which gcc` 输出路径 |

### 指标 c:`C:/msys64/mingw64/bin` 已加入 PATH

| 字段 | 内容 |
|------|------|
| **检查项** | 步骤 `Verify MinGW toolchain (Windows)`(release.yml L149-156)中 `which gcc` / `which ar` 输出路径 |
| **预期结果** | 同时满足:<br>1. `which gcc` 输出 `/c/msys64/mingw64/bin/gcc`(非 "gcc not in PATH")<br>2. `which ar` 输出 `/c/msys64/mingw64/bin/ar`(非 "ar not in PATH") |
| **失败处理** | 见 §4.3。若 `which gcc` 返回其他路径(如 `/usr/bin/gcc`),说明 PATH 优先级被覆盖,需排查 `GITHUB_PATH` 注入时机 |

### 指标 d:`gcc --version` 和 `ar --version` 可执行

| 字段 | 内容 |
|------|------|
| **检查项** | 同指标 c 步骤中 `gcc --version` / `ar --version` 输出 |
| **预期结果** | 同时满足:<br>1. `gcc --version` 输出形如 `gcc.exe (Revxxxx, Built by MSYS2 project) 14.x.x`<br>2. `ar --version` 输出形如 `GNU ar 2.42` |
| **失败处理** | 见 §4.3。`--version` 报错说明 MSYS2 安装不完整,可在 `setup-msys2` 步骤增加 `update: true` 强制刷新包数据库 |

### 指标 e:`cargo build --release --target x86_64-pc-windows-gnu -p chimera-cli` 成功

| 字段 | 内容 |
|------|------|
| **检查项** | 步骤 `Build binary`(release.yml L183-210)是否成功退出 |
| **预期结果** | 步骤退出码 0;日志末尾出现 `Finished \`release\` profile [optimized] target(s) in X.YZs`;无 `error: linking with cc failed` / `cannot find gcc` / `undefined reference` |
| **失败处理** | 见 §4.4。失败时本步骤自动通过 `::error::` annotation 输出前 30 条 error,并在日志末尾打印 `tail -n 100 build.log`;同时触发 `Upload build log on failure` 步骤上传 `build.log` artifact(保留 7 天) |

**关键诊断区**:
- `=== Rust toolchain info ===` 段确认 `stable-x86_64-pc-windows-gnu`
- `=== MinGW toolchain info ===` 段确认 gcc/ar 路径
- `=== Build start ===` 后为实际编译输出

### 指标 f:binary `--version` 输出匹配正则

| 字段 | 内容 |

**备注**:binary 内部 codename 为 `aether`(`crates/chimera-cli/Cargo.toml [[bin]]`),CI 重命名为 `chimera`;二者均符合正则。

### 指标 g:binary 体积 < 50MB

| 字段 | 内容 |
|------|------|
| **检查项** | 步骤 `Verify binary size < 50MB`(release.yml L239-251)输出 |

## 4. 失败处理流程

### 4.1 通用流程

1. **下载日志**:在失败 job 页面点击右上角 `...` → `Download logs`,保存完整日志便于离线分析
2. **下载 build.log**(仅指标 e 失败时):在 workflow run 详情页 Artifacts 区下载 `build-log-x86_64-pc-windows-gnu`,内含完整 `cargo build` 输出(含警告与错误)
3. **定位失败指标**:按 §3 顺序对照日志,确认首个失败的指标编号
4. **执行对应诊断**:参见 §4.2-4.7 常见错误诊断,深度根因分析见姐妹文档 `v1.5.1-omega_ci_windows_gnu_verification.md` §4.2-4.8
5. **修复后重测**:修复后删除已推送的 tag(`git tag -d v1.5.1-omega && git push origin :refs/tags/v1.5.1-omega`),重新打 tag 推送触发 CI
6. **记录教训**:将根因与修复方案追加到本文件 §6 "验证记录",必要时同步到 `project_memory.md`

### 4.2 指标 a 失败(msys2/setup-msys2@v2 安装失败)

**常见错误诊断**:
- 日志含 `Failed to install` / `network error`:网络瞬时失败或 MSYS2 镜像源不可达
- 日志含 `gcc package not found`:MSYS2 包数据库未刷新或 action 版本回归
- `windows-latest` runner 当日公告:https://www.githubstatus.com/

**修复方向**:
- 网络瞬时失败:重试(重新打 tag 推送)
- action 版本回归(`@v2` release tag 被 upstream 污染):临时锁定到具体 commit SHA,如 `msys2/setup-msys2@<sha>`
- MSYS2 镜像问题:在 `setup-msys2` 步骤添加 `update: false` 跳过首次更新(牺牲版本新鲜度换取稳定性)

### 4.3 指标 c/d 失败(Verify MinGW toolchain 输出异常)

**常见错误诊断**:
- `which gcc` 返回空或 "not in PATH":`GITHUB_PATH` 注入未生效或被后续步骤覆盖
- `which gcc` 返回非 `C:/msys64/mingw64/bin` 路径(如 `/usr/bin/gcc`):PATH 优先级问题,runner 预装的其他 gcc 抢先
- `gcc --version` 报错:MSYS2 安装不完整或 gcc 包未正确安装

**修复方向**:
- PATH 未生效:在 `Verify MinGW toolchain` 步骤前显式 `export PATH="/c/msys64/mingw64/bin:$PATH"`(临时验证,长期需排查 `GITHUB_PATH` 时机)
- 路径冲突:确认 `Configure MinGW linker for CI` 步骤的 `echo "C:/msys64/mingw64/bin" >> $GITHUB_PATH` 无重复 PATH 注入覆盖
- gcc 损坏:在 `setup-msys2` 步骤增加 `update: true` 强制刷新 MSYS2 包数据库

### 4.4 指标 e 失败(Build binary 构建失败)

**常见错误诊断**(按错误类型区分):
- `linking with cc failed` / `cannot find gcc`:linker 路径问题(回看指标 c/d)
- `cannot find ar` / `windres`:PATH 未注入 `C:/msys64/mingw64/bin`(回看指标 b/c)
- `undefined reference to '_imp__...'`:MinGW 与 MSVC 运行时混用,检查是否有 crate 误链 MSVC 库(如 `windows-sys` 某些 feature)
- `error[E0xxx]`:纯 Rust 编译错误,与 MinGW 无关,本地应已捕获
- `STATUS_STACK_BUFFER_OVERRUN (0xC0000409)`:实际为 OOM(`__fastfail` 的 `FAST_FAIL_FATAL_APP_EXIT`),参考 §7.1 `--jobs 2` 缓解

**修复方向**:
- linker 问题:回到 §4.3 修复 MinGW 配置
- 运行时混用:检查 `Cargo.toml` 是否引入了 `windows-msvc` 专属依赖,改为 target-specific 依赖
- OOM:在 `Build binary` 步骤添加 `CARGO_BUILD_JOBS=2` 环境变量(与本地 `--jobs 2` 一致)
- 纯编译错误:本地修复后重新打 tag

### 4.5 指标 f 失败(--version 输出异常)

**常见错误诊断**:
- 输出为空:binary 可能 panic 在启动阶段,需下载 artifact 本地运行 `./chimera-windows-x86_64.exe --version` 确认
- 版本号不匹配:`Cargo.toml` `workspace.package.version` 与 tag 不一致(§7.2 第 9 项检查遗漏)
- "version check skipped":binary 退出码非 0,需查看 stderr 输出

**修复方向**:
- 版本号不匹配:更新 `Cargo.toml` `workspace.package.version` 为 tag 对应版本,重新打 tag
- binary panic:下载 artifact 本地用 `RUST_BACKTRACE=1` 运行定位 panic 位置
- 缺失子命令:检查 `chimera-cli` 的 clap 配置是否正确暴露 `--version`

### 4.6 指标 g 失败(Binary 体积超 50MB)

**常见错误诊断**:
- 确认实际体积(日志中 `Binary size: <N> bytes`)
- 排查近期是否引入重量级依赖(`Cargo.lock` diff)
- 确认 release profile 配置未失效(`Cargo.toml` `[profile.release]` 仍为 `lto=true` + `opt-level="z"` + `strip=true`)

**修复方向**:
- 依赖膨胀:评估新依赖必要性,或拆分为 feature gate
- profile 失效:恢复 release profile 配置
- 临时放宽阈值:仅在确认体积增长合理(如新增核心功能)且经评审后,可临时调高 `release.yml` L247 的阈值(不推荐,应优先优化体积)

### 4.7 指标 h 失败(Artifact 上传失败)

**常见错误诊断**:
- `artifact name conflict`:matrix 中存在重名 artifact(检查 `artifact_name` 字段唯一性)
- `upload failed`:runner 与 GitHub artifact 存储之间的网络问题
- `retention-days invalid`:`retention-days: 30` 配置无效(应为 1-90 整数)

**修复方向**:
- 名称冲突:修正 matrix `artifact_name` 字段
- 网络问题:重试
- retention 配置:确认 `retention-days: 30` 在合法范围
