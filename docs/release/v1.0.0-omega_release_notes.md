## 1. 发布概要

Chimera CLI v1.0.0-omega 是 NEXUS-OMEGA 项目历经 **8 周开发周期** 完成的首个生产就绪版本。本版本实现 **34 个 crate**(覆盖 10 层架构 L1→L10),累计 **3002+ 测试** 全绿,严格遵循 **OMEGA 四定律**(Ω-Sparse / Ω-Compress / Ω-Evolve / Ω-Event),融合 **37 个创新点**(22 个第一代 + 15 个第三代),将 Claude Code 尸检教训、Hermes 基因、Qoder 骨骼与五大模型灵魂整合为统一的工程实体。

本版本的核心价值:

- **跨平台可达**:5 平台 matrix CI/CD(Windows/Linux/macOS × x86_64/aarch64),Docker distroless 镜像 < 100MB
- **文档完备**:README 重写、CODE_WIKI 34 crate 全覆盖、架构文档 4 个、cargo doc 零 warning、cargo fmt 零 diff

## 4. 安全特性

### 4.1 `#![forbid(unsafe_code)]` 40/40 全覆盖

所有 34 个 workspace crate + 6 个附属 crate(fuzz / benches 等)均声明 `#![forbid(unsafe_code)]`,这是项目最硬的安全红线。Week 8 验证 40/40 覆盖,无任何 crate 放松约束。

**关键决策**:rusqlite `bundled` feature 与 `forbid(unsafe_code)` 兼容——`forbid` 是 crate 级,只扫描当前 crate 源码,不传播到依赖,因此 SQLite 持久化存储可用于 mlc-engine / quest-engine。

### 4.2 OWASP Top 10 渗透测试(20/20 通过)

`tests/security/owasp_top10.rs` 覆盖 A01-A10,每项含正常用例 + 攻击用例:

- SecCore 零信任沙箱拒绝未知命令(A04 Insecure Design)
- 注入字符(`;` `|` `$()` 等)被 SecCore 过滤(A03 Injection)
- 跨层通信仅走 EventBus,无向上依赖(A08 Software & Data Integrity)
- AHIRT 反黑客红队 5 分钟周期审计(A09 Security Logging)

### 4.3 模糊测试(cargo-fuzz 3 target)

`fuzz/` 独立 crate(不污染主 workspace),3 target:

- `quest_parse`:Quest 解析器模糊

> **限制**:`cargo fuzz` 要求 nightly toolchain,3 target 已就绪待 CI 环境 nightly 完整运行。

### 4.4 依赖审计(cargo-audit 无高危)

### 6.1 Windows clippy 并行编译栈溢出(STATUS_STACK_BUFFER_OVERRUN)

> **根因分析完成(2026-06-27,Week 8 限制深度攻坚 Task 1)**:经 procdump + WER minidump + objdump 反汇编定位,实际根因为 **OOM(堆内存分配失败)**,而非栈问题。崩溃函数为 `std::alloc::rust_oom`,通过 `__fastfail(FAST_FAIL_FATAL_APP_EXIT=7)` 终止进程;`STATUS_STACK_BUFFER_OVERRUN (0xC0000409)` 是 `__fastfail` 的统一异常代码(误导性命名)。详见 `docs/dev/clippy_root_cause_analysis.md`(7 章节完整报告)+ `docs/dev/upstream_clippy_issue_draft.md`(上游 issue 草稿)。

**现象**:Windows GNU 工具链下,`cargo clippy --workspace --all-targets`(默认并行 jobs = CPU 核数)时,多个 `clippy-driver.exe` 进程崩溃,exit code `0xc0000409`(STATUS_STACK_BUFFER_OVERRUN),连锁导致 E0786 invalid metadata / E0463 can't find crate / internal compiler error,最终编译失败(exit 101)。

**已尝试方案(Week 8 Task 3 根因分析)**:

- `RUST_MIN_STACK=33554432`(32MB 栈):**无效**。实验 C(默认 jobs + 32MB 栈)仍触发 STATUS_STACK_BUFFER_OVERRUN(exit 101, 89.75s)
- `CARGO_INCREMENTAL=0`:配合 `--jobs` 使用,本身不解决栈溢出

**对比实验表格**(均设置 `RUST_MIN_STACK=33554432` + `CARGO_INCREMENTAL=0`,全新 `CARGO_TARGET_DIR` 从零编译):

| 实验 | --jobs | 结果 | 耗时 | STATUS_STACK_BUFFER_OVERRUN | 警告数 |
|------|--------|------|------|------------------------------|--------|
| A | 1(串行) | ✅ exit 0 | 600.69s(10m00s) | 无 | 0 |
| B | 2(低并行) | ✅ exit 0 | 335.97s(5m36s) | 无 | 0 |
| C | 默认(CPU 核数) | ❌ exit 101 | 89.75s(崩溃) | 有(0xc0000409) | N/A |

**推荐 workaround**(`--jobs 2`,比 `--jobs 1` 快 44% 且避免栈溢出):

```powershell
$env:RUST_MIN_STACK = '33554432'       # 32MB 栈(保险措施,虽非根因解)
$env:CARGO_INCREMENTAL = '0'           # 禁用增量编译,减少文件锁竞争
cargo clippy --workspace --all-targets --jobs 2 -- -D warnings
```

**根因分析(Week 8 限制深度攻坚 Task 1,2026-06-27 完成)**:经 procdump + WER minidump + objdump 反汇编四重互证,实际根因为 **OOM(堆内存分配失败)**,而非栈问题。捕获过程:procdump v12.0(下载自 Sysinternals)以 `-ma -e 1` 循环监控 40+ 个 clippy-driver PID,但**未能捕获 dump**——因为 `__fastfail`(`int 0x29`)绕过 SEH 异常分发,procdump 异常钩子不触发;转而依赖 WER 默认配置,在 `C:\Users\...\AppData\Local\CrashDumps\` 自动生成 4 个 minidump(~2.6MB 各)。

MINIDUMP 异常流解析:ThreadId 34388,ExceptionCode `0xC0000409`,ExceptionAddress `0x7FFA08CF71B1`(低 32 位 = std DLL RVA `0x171B1`)。WER BEX64 报告 P9 = `0x7` = `FAST_FAIL_FATAL_APP_EXIT`。

objdump 反汇编 `std-b0558c7fd7f3aef7.dll` 在 RVA `0x171B1` 处的指令为 `mov $0x7,%ecx; int $0x29`(`__fastfail(7)`),所属函数符号为 `_RNCNvNtCs1ol9KfofPpO_3std5alloc8rust_oom0B5_` = **`std::alloc::rust_oom`**(Rust 全局分配器的 OOM 处理函数)。`rust_oom` 仅在 `GlobalAlloc::alloc()` 返回 null 时被调用,进入此函数 ⟺ 发生 OOM。`0xC0000409 (STATUS_STACK_BUFFER_OVERRUN)` 是 `__fastfail` 的**统一异常代码**(无论 fastfail code 2/7/14 均显示为此),异常名具有误导性——必须看 fastfail code(P9=7)才能区分。

排除的假设:栈空间不足(应为 `0xC00000FD`,且 RUST_MIN_STACK=32MB 无效)、`/GS` 栈金丝雀失败(fastfail code 应为 14 或 2,且崩溃函数应为 `__stack_chk_fail` 而非 `rust_oom`)、堆损坏(fastfail code 应为 21)、clippy 业务逻辑 bug(故障模块是 std DLL,固定偏移)。

触发机制(推断):默认 `--jobs` = CPU 核数,多个 clippy-driver 并行,各自加载 `rustc_driver-*.dll` + `std-*.dll` + 5 个 proc-macro DLL(thiserror_impl/serde_derive/zerocopy_derive/tokio_macros/tracing_attributes),proc-macro 在进程内执行,堆压力叠加 → 某 `alloc()` 返回 null → `handle_alloc_error` → `rust_oom` → `__fastfail(7)` → 进程终止 → 连锁 `.rmeta` 不完整导致 E0786/E0463。

**后续计划**:
- 向 rust-lang/rust-clippy 上游提交 issue(草稿已就绪:`docs/dev/upstream_clippy_issue_draft.md`,8 章节含复现步骤、minidump 分析、反汇编证据、3 组对比实验、workaround、根因推断)
- CI 环境(GitHub Actions Linux runner)不受影响(Linux 无 `__fastfail` 机制,且有 cgroup 内存保护)
- 本机 Windows 开发继续采用 `--jobs 2` workaround(根因解需上游修复或迁移 MSVC 工具链)
- 可选长期改进:迁移 `stable-x86_64-pc-windows-msvc`(std 静态链接,无 DLL 共享状态)

**深度攻坚完成(2026-06-27)**:限制 5 clippy 根因分析突破性完成(OOM,非栈),限制 1/2/3 通过 CI workflow 就绪 + tag 触发实际执行,3 项限制全部从"部分解除/委托 CI"升级为"完全解除"或"根因分析完成"。详见 `docs/acceptance/week8_limitations_deep_remediation_report.md`。

### 9.3 GA 版本号决策(2026-07-04)

- 缓解:用户安装时使用 install.sh / install.ps1 默认拉取 master 分支最新版本,可获取最新修复
