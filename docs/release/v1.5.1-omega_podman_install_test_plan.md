## 1. 测试计划总览

### 1.1 测试目标

按安装指南 §1-§9 的章节顺序,逐项验证 Podman 在 Chimera CLI 开发环境中的可用性,确保:

1. **安装路径可达性** — winget / MSI / Chocolatey 三种安装方式至少一种可跑通
2. **Machine 配置可重复性** — `podman machine init/start/ls` 行为符合预期,资源参数生效
3. **Docker CLI 兼容性** — `Set-Alias docker podman` 后,所有 `docker <subcommand>` 透明走到 podman
4. **项目脚本对齐性** — `scripts/verify_docker_locally.ps1` 的 Podman 路径(`Invoke-EngineVerification -Engine "podman"`)与 CI `release.yml` docker job 在验证维度上完全等价
5. **故障场景可恢复性** — WSL2 异常 / 网络代理 / 镜像存储问题有明确排查路径

### 1.2 测试范围与不测范围

| 范围 | 是否测试 | 说明 |
|------|---------|------|
| Windows 11 + WSL2 后端 | ✅ 测试 | 主开发环境(对齐 §1.1) |
| Linux 原生安装 | ⚠️ 仅冒烟 | 安装指南 §10.1 简述,本计划不展开 |
| macOS 安装 | ⚠️ 仅冒烟 | 安装指南 §10.2 简述,本计划不展开 |
| Podman Desktop GUI | ❌ 不测 | 安装指南 §4.1 可选项,CLI 验证已足够覆盖项目脚本 |
| Kubernetes Pod 功能 | ❌ 不测 | Chimera CLI 项目无 K8s 集成需求 |
| `podman compose` 子命令 | ❌ 不测 | 项目不使用 compose 编排 |

### 1.3 测试用例编号规则

```
TC-<章节号>-<用例序号>
例如:
  TC-02-01  →  第 2 章(环境前置条件)第 1 个用例
  TC-03-02  →  第 3 章(安装步骤)第 2 个用例
  TC-06-03  →  第 6 章(verify_docker_locally.ps1 验证)第 3 个用例
```

### 1.4 测试通过/失败/阻塞判定

| 状态 | 判定标准 |
|------|---------|
| ✅ 通过 | 实际输出符合"预期输出"字段,验证检查点全部命中 |
| ❌ 失败 | 实际输出与"预期输出"不符,或验证检查点存在未命中项 |
| ⛔ 阻塞 | 前置依赖未满足(如 WSL2 不可用、网络完全不可达),无法执行该用例 |
| ⏭ 跳过 | 该安装方式不适用当前环境(如未安装 Chocolatey 则跳过 TC-03-03) |

### 1.5 测试执行顺序

按依赖关系串行执行,失败用例不阻塞后续独立用例:

## 2. 测试环境前置条件

### 2.1 系统要求核对

#### TC-02-01:Windows 版本与架构核对

**测试步骤**:

```powershell
# 1. 查看 Windows 版本
[System.Environment]::OSVersion.Version
# 或
winver  # GUI 查看

# 2. 查看操作系统详细信息
Get-CimInstance -ClassName Win32_OperatingSystem |
    Select-Object Caption, Version, BuildNumber, OSArchitecture

# 3. 查看处理器架构
$env:PROCESSOR_ARCHITECTURE
```

**预期输出**:

| 字段 | 预期值 | 实测值 |
|------|--------|--------|
| `Caption` | `Microsoft Windows 11 Pro`(或 Home/Enterprise) | (待填) |
| `BuildNumber` | ≥ 22000 | (待填) |
| `OSArchitecture` | `64-bit` | (待填) |
| `PROCESSOR_ARCHITECTURE` | `AMD64` | (待填) |

**验证检查点**:

- [ ] Windows 11 Build 22000+ (Windows 10 Build 19043+ 也接受,但需备注)
- [ ] 架构为 AMD64 / x86_64
- [ ] 若为 ARM64,标记为 ⛔ 阻塞(本测试计划未覆盖 ARM64)

#### TC-02-05:资源与磁盘空间检查

**测试步骤**:

```powershell
# 1. 物理内存
(Get-CimInstance -ClassName Win32_ComputerSystem).TotalPhysicalMemory / 1GB

# 2. 系统盘可用空间
Get-PSDrive C | Select-Object Name, @{Name="FreeGB";Expression={[math]::Round($_.Free/1GB, 2)}}

# 3. D 盘可用空间(若存在)
Get-PSDrive D -ErrorAction SilentlyContinue | Select-Object Name, @{Name="FreeGB";Expression={[math]::Round($_.Free/1GB, 2)}}

# 4. CPU 核数
(Get-CimInstance -ClassName Win32_Processor).NumberOfLogicalProcessors
```

**预期输出**:

| 字段 | 预期值 | 实测值 |
|------|--------|--------|
| 物理内存 | ≥ 6 GB(推荐 8 GB+) | (待填) |
| C 盘可用空间 | ≥ 20 GB | (待填) |
| D 盘可用空间 | ≥ 60 GB(若用于镜像存储迁移) | (待填) |
| CPU 逻辑核数 | ≥ 4(推荐) | (待填) |

**验证检查点**:

- [ ] 物理内存 ≥ 6 GB
- [ ] C 盘或 D 盘至少一处可用空间 ≥ 20 GB
- [ ] 若内存 < 6 GB:标记为 ⚠️ 警告,Podman Machine 默认配置可能 OOM
- [ ] 若磁盘 < 20 GB:标记为 ❌ 失败,WSL2 虚拟硬盘无法创建

#### TC-03-04:winget 安装失败的回退(官方 MSI)

```powershell
Invoke-WebRequest -Uri $msiAsset.browser_download_url -OutFile "$env:TEMP\podman-setup.exe"

# 2. 静默安装
Start-Process "$env:TEMP\podman-setup.exe" -ArgumentList "/S" -Wait

# 3. 重新打开 PowerShell

# 4. 验证
podman --version
```

**预期输出**:

```text

podman version 5.x.x
```

**验证检查点**:

- [ ] MSI 安装过程无 error 对话框

## 4. Podman Machine 配置测试用例

Podman 在 Windows 上通过 "Podman Machine"(WSL2 Linux VM)运行容器,首次使用前必须初始化。

### 4.1 默认配置初始化

#### TC-04-01:podman machine init(默认配置)

**测试步骤**:

```powershell
# 1. 初始化默认 machine(默认:2 CPU / 2048 MB RAM / 20 GB 磁盘)
podman machine init

# 2. 等待初始化完成(首次会拉取 Fedora CoreOS 镜像,可能需要 5-15 分钟,取决于网络)

# 3. 验证 machine 已创建
podman machine list
```

### 9.4 测试中常见疑问解答

**Q1:为什么 TC-06-01 要用 `powershell -NoProfile` 模式?**

A:因为在加载 Profile 的会话中,`Set-Alias docker podman` 会使 `Get-Command docker` 返回非空,导致脚本进入路径 1(Docker 验证)而非路径 2(Podman 验证)。`-NoProfile` 模式不加载用户 Profile,确保 `docker` 命令真的不可用,从而触发脚本的 Podman 路径。

**Q2:为什么 TC-06-07 中本地未注入 VERSION build-arg?**

A:`verify_docker_locally.ps1` 的 `Invoke-EngineVerification` 函数 L107 执行 `& $Engine build -t $ImageName $RepoRoot`,未传 `--build-arg VERSION=...`,因此使用 Dockerfile L49 的默认值 `ARG VERSION=1.5.1-omega`。CI 通过 `docker/build-push-action@v5` 的 `build-args: VERSION=${{ github.ref_name }}` 注入实际 tag。本地验证镜像构建配方正确性,版本号注入路径由 CI 独立验证,二者互补不冲突。

**Q3:测试中发现 Podman 行为与安装指南描述不符怎么办?**

A:按以下顺序处理:
1. 在测试报告(§8.3)中详细记录实际行为
2. 在 GitHub issue 中提交差异描述,附测试输出
3. 评估是否需要修订 `v1.5.1-omega_podman_setup_guide.md`(由文档维护者决策)
4. 若差异影响项目脚本(`verify_docker_locally.ps1`)的正确性,需 ADR 记录并修复脚本

**Q4:测试报告应保存到哪里?**

A:推荐保存到 `docs/release/v1.5.1-omega_podman_install_test_report_<YYYYMMDD>.md`,与测试计划文件同目录,便于追溯。文件名中的日期使用测试执行日期。

**Q5:测试通过后是否需要更新 `nuxus规则.md` §10.5?**

A:不需要直接更新。§10.5 的 "Podman 安装指南文档已存在但尚未实际安装测试" 短板属于文档级短板,本测试计划通过后,可在 `project_memory.md` 记录 "Podman 安装验证路径已通过测试,可作为 Docker Desktop 替代",作为运维知识保留。若发现需修改脚本代码,则按 ADR 流程处理。
