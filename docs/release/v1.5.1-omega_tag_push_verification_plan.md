## 1. 背景与目标

### 1.1 §10.5 P1 短板

`.trae/rules/nuxus规则.md` §10.5 登记的 P1 短板:

| 短板 | 影响 | 优先级 | 状态 |
|------|------|--------|------|
| release.yml Windows job 未安装 MinGW | `windows-latest` runner 默认无 `D:/msys64/...`,GNU target 可能链接失败 | P1 | 🔧 待下次 tag 推送验证(已配置 msys2/setup-msys2,见验证清单) |

**根因**:`release.yml` build job 的 Windows matrix 使用 `x86_64-pc-windows-gnu` target,需要 `gcc.exe` 链接器;但 `windows-latest` runner 默认不安装 MinGW,而本地 `.cargo/config.toml` 硬编码指向 `D:/msys64/mingw64/bin/gcc.exe`(本地 D 盘工具链)。

**已有修复**:`release.yml` L129-147 已通过三步配置覆盖本地路径:
1. L129-135 `msys2/setup-msys2@v2` 安装 MSYS2 + gcc
2. L137-147 `CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER` 环境变量覆盖 linker 路径 + `GITHUB_PATH` 注入
3. L149-156 软验证 `which gcc` / `gcc --version`

**为何仍需验证**:build job 仅由 tag 触发(L47 `if: startsWith(github.ref, 'refs/tags/v')`),PR 触发只跑 test job(L269),因此 Windows GNU 的 MinGW 配置从未在真实 CI 环境中跑通。

### 1.2 release.yml 触发机制

| 触发方式 | 触发条件 | 运行的 job |
|----------|---------|-----------|
| `push` tag | `v*.*.*-omega`(如 `v1.5.1-omega`) | build + test + docker + deb + release |
| `pull_request` | 改动 `Cargo.toml`/`Cargo.lock`/`crates/**`/`.github/workflows/**` | 仅 test(L269 条件) |
| `workflow_dispatch` | 手动触发 | 仅 test(build/docker/deb/release 的 `if` 均要求 tag) |

### 1.3 5 个 job 概览

| Job | 行号 | runner | 触发条件 | 作用 |
|-----|------|--------|---------|------|
| `build` | L42-258 | 5 平台 matrix | tag | 编译 5 平台 release binary |
| `test` | L263-310 | ubuntu-latest | tag + PR | 全 workspace 测试 |
| `docker` | L315-396 | ubuntu-latest | tag | 构建镜像 + 推送 GHCR + 验证 |
| `deb` | L400-428 | ubuntu-latest | tag | 构建 Debian .deb 包 |
| `release` | L433-512 | ubuntu-latest | tag | 汇总产物 + 创建 GitHub Release |

**job 依赖关系**(L317 `needs: build` / L435 `needs: [build, test, docker, deb]`):

```
build (5 平台并行) ─┬─→ docker ─┐
                    ├─→ deb ────┤
                    └───────────┼─→ release
test ─────────────────────────────┘
```

`fail-fast: false`(L49)确保单平台 build 失败不阻塞其他平台,但 `release` job 依赖全部 build 完成(L435 `needs: [build, ...]`),任何 build 失败会导致 release job 跳过。

### 1.4 验证目标

1. **P1 重点**:确认 Windows GNU target 在 `windows-latest` runner 上通过 MSYS2 配置能成功完成 release 构建
2. **全平台覆盖**:确认 5 平台 binary 全部产出且符合契约(体积 < 50MB、`--version` 输出匹配正则)
3. **Docker 镜像**:确认 GHCR 镜像体积 < 100MB、`--version` 功能正常、可公开拉取
4. **Release 完整性**:确认 GitHub Release 附件数量 = 6(5 binary + checksums.txt + .deb,实际 checksums 含 6 项),正文符合预期
5. **fuzz.yml**:确认 6 个 fuzz target 并行运行(非阻塞,crash 上传 90 天留存)

## 3. 环境配置详情:CI 与本地 MinGW 差异分析

### 3.1 本地环境配置(Windows 11 + D 盘工具链)

| 配置项 | 值 | 来源文件 |
|--------|-----|---------|
| Rust 工具链 channel | `stable-x86_64-pc-windows-gnu` | `rust-toolchain.toml` |
| gcc 路径 | `D:/msys64/mingw64/bin/gcc.exe` | `.cargo/config.toml` `[target.x86_64-pc-windows-gnu] linker` |
| incremental | `false` | `.cargo/config.toml` `[build] incremental` |
| SQLITE_ENABLE_FTS5 | `"1"` | `.cargo/config.toml` `[env]` |
| CARGO_HOME | `D:\Chimera CLI\.toolchain\cargo` | 用户环境变量(见 `.claude/CLAUDE.md` §1) |
| RUSTUP_HOME | `D:\Chimera CLI\.toolchain\rustup` | 用户环境变量 |

### 3.2 CI 环境配置(windows-latest runner)

| 配置项 | 值 | 来源 |
|--------|-----|------|
| Rust 工具链 channel | `stable-x86_64-pc-windows-gnu` | `release.yml` L58 matrix.toolchain + L93-97 dtolnay/rust-toolchain |
| MSYS2 安装路径 | `C:/msys64` | `msys2/setup-msys2@v2` 默认(L131-135) |
| gcc 路径 | `C:/msys64/mingw64/bin/gcc.exe` | `CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER` 环境变量(L144) |
| PATH 注入 | `C:/msys64/mingw64/bin` | `GITHUB_PATH`(L147) |
| incremental | `0`(禁用) | `release.yml` L32 `CARGO_INCREMENTAL: 0` |
| RUST_BACKTRACE | `1` | `release.yml` L36 |

### 3.3 配置差异对比表

| 维度 | 本地 | CI | 差异处理 |
|------|------|-----|---------|
| gcc 路径 | `D:/msys64/mingw64/bin/gcc.exe` | `C:/msys64/mingw64/bin/gcc.exe` | 盘符不同(D vs C),CI 通过环境变量覆盖 |
| 覆盖机制 | `.cargo/config.toml` 直接配置 | `CARGO_TARGET_<TRIPLE>_LINKER` 环境变量 | 环境变量优先级 > `.cargo/config.toml`,无需修改仓库文件 |
| ar/windres 路径 | PATH 中已有 `D:/msys64/mingw64/bin` | `GITHUB_PATH` 注入 `C:/msys64/mingw64/bin` | cargo 链接阶段需要 ar/windres,仅设 linker 不足 |
| CARGO_HOME | `D:\Chimera CLI\.toolchain\cargo` | runner 默认 `~/.cargo` | CI 不使用项目内工具链,cargo registry 缓存由 actions/cache@v4 管理 |
| 工具链安装方式 | 项目内 `.toolchain/` | dtolnay/rust-toolchain@master action | CI 由 action 安装到 runner 默认位置 |

### 3.4 覆盖机制原理

**环境变量优先级链**:

```
1. CARGO_TARGET_<TRIPLE>_LINKER 环境变量(最高优先级,CI 使用)
2. .cargo/config.toml [target.<triple>] linker(本地使用)
3. cargo 内置默认(回退)
```

**关键设计**:`release.yml` L144 通过 `echo "CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER=C:/msys64/mingw64/bin/gcc.exe" >> $GITHUB_ENV` 设置环境变量,优先级高于 `.cargo/config.toml` 的 `D:/msys64/...`,因此 CI 无需修改仓库文件即可覆盖 linker 路径。

**GITHUB_PATH vs GITHUB_ENV**:
- `GITHUB_ENV`:设置环境变量,后续 step 可读
- `GITHUB_PATH`:追加到 PATH,后续 step 的命令查找会包含此路径
- L147 注入 `C:/msys64/mingw64/bin` 到 PATH,是为了让 cargo 能找到 `ar`、`windres` 等 MinGW 辅助工具(仅设置 linker 不足,链接阶段还需要这些工具)

### 3.5 5 平台 runner 工具链差异表

| 平台 | Target | Runner | 工具链 | 交叉编译 | 特殊依赖 |
|------|--------|--------|--------|---------|---------|
| Windows x86_64 | `x86_64-pc-windows-gnu` | `windows-latest` | `stable-x86_64-pc-windows-gnu`(L58) | 否 | MSYS2 + MinGW(L129-147) |
| Linux x86_64 | `x86_64-unknown-linux-gnu` | `ubuntu-latest` | `stable`(L96 回退) | 否 | `libsqlite3-dev` `pkg-config`(L120-124) |
| Linux aarch64 | `aarch64-unknown-linux-gnu` | `ubuntu-latest` | `stable` + `cross` | 是(L70 `cross: true`) | `cargo install cross --locked`(L171-173) |
| macOS x86_64 | `x86_64-apple-darwin` | `macos-latest` | `stable` | 否 | 无 |
| macOS aarch64 | `aarch64-apple-darwin` | `macos-latest` | `stable` | 否 | 无 |

**非 Windows 平台的工具链覆盖**(L103-106):通过 `RUSTUP_TOOLCHAIN=stable` 环境变量强制使用 stable,避免 `rust-toolchain.toml` 的 `stable-x86_64-pc-windows-gnu` channel 在 Linux/macOS runner 上导致 toolchain 选择冲突。

### 4.1 build job 验证(5 平台 matrix)

build job 包含 5 个并行的 matrix 实例,每个实例运行相同的 steps 序列。**重点验证 Windows GNU 实例**(P1 短板),其他 4 平台做基本完整性验证。

#### 4.1.1 Windows x86_64-pc-windows-gnu(P1 重点验证)

> 点击 `Build (x86_64-pc-windows-gnu)` job 进入详细日志,按以下 step 顺序核验。

**Step 1: Checkout**(L87-88)
- 预期:退出码 0,无特殊输出
- 失败:极少见,通常是 runner 临时故障,重试即可

**Step 2: Setup Rust toolchain**(L90-97)
- 预期:安装 `stable-x86_64-pc-windows-gnu` toolchain,日志含 `rustup toolchain install stable-x86_64-pc-windows-gnu`
- 失败:网络问题导致 rustup 下载失败,重试

**Step 3: Ensure Windows GNU toolchain is default**(L111-116)
- 预期:`rustup default stable-x86_64-pc-windows-gnu` 成功,`rustup show` 输出显示 default toolchain 为 GNU
- 关键日志:`Default toolchain set to 'stable-x86_64-pc-windows-gnu'`

**Step 4: Setup MSYS2 and MinGW (Windows GNU target)**(L129-135)— **P1 关键指标 1**
- 预期:退出码 0;日志含 `MSYS2 installed` / `Installing package gcc` 且无 error
- 首次运行下载 MSYS2 完整镜像(约 200MB),耗时 3-5 分钟属正常
- 失败处理:见 §7.2 指标 1

**Step 5: Configure MinGW linker for CI**(L137-147)— **P1 关键指标 2**
- 预期:退出码 0;`$GITHUB_ENV` 写入 `CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER=C:/msys64/mingw64/bin/gcc.exe`;`$GITHUB_PATH` 写入 `C:/msys64/mingw64/bin`
- 本 step 日志较短,主要看退出码;真正生效需看 Step 6 的 `which gcc` 输出路径
- 失败处理:见 §7.2 指标 2

**Step 6: Verify MinGW toolchain (Windows)**(L149-156)— **P1 关键指标 3+4**
- 预期同时满足:
  1. `which gcc` 输出 `/c/msys64/mingw64/bin/gcc`(非 "gcc not in PATH")
  2. `which ar` 输出 `/c/msys64/mingw64/bin/ar`(非 "ar not in PATH")
  3. `gcc --version` 输出形如 `gcc.exe (Revxxxx, Built by MSYS2 project) 14.x.x`
  4. `ar --version` 输出形如 `GNU ar 2.42`
- 若 `which gcc` 返回其他路径(如 `/usr/bin/gcc`),说明 PATH 优先级被覆盖,需排查 `GITHUB_PATH` 注入时机
- 失败处理:见 §7.2 指标 3+4

**Step 7: Cache cargo registry & target**(L158-167)
- 预期:首次运行 `Cache not found`,后续 run `Cache restored from key`
- 失败:缓存层故障不影响构建,仅影响速度,可忽略

**Step 8: Build binary**(L183-210)— **P1 关键指标 5**
- 预期:退出码 0;日志末尾出现 `Finished \`release\` profile [optimized] target(s) in X.YZs`;无 `error: linking with cc failed` / `cannot find gcc` / `undefined reference`
- 关键诊断区:
  - `=== Rust toolchain info ===` 段确认 `stable-x86_64-pc-windows-gnu`
  - `=== MinGW toolchain info ===` 段确认 gcc/ar 路径
  - `=== Build start ===` 后为实际编译输出
- 失败处理:见 §7.2 指标 5;失败时本 step 自动通过 `::error::` annotation 输出前 30 条 error,并在日志末尾打印 `tail -n 100 build.log`;同时触发 Step 9 上传 `build.log` artifact(保留 7 天)

**Step 9: Upload build log on failure**(L212-218)
- 仅在 Step 8 失败时执行
- 预期:上传 `build-log-x86_64-pc-windows-gnu` artifact 到 workflow run
- 失败:若此 step 也失败,需 Download job logs 获取完整日志

**Step 10: Rename to artifact name**(L228-232)
- 预期:`cp target/x86_64-pc-windows-gnu/release/aether.exe chimera-windows-x86_64.exe` 成功
- 失败:通常是 Step 8 构建失败导致 binary 不存在,回看 Step 8

**Step 11: Verify binary runs**(L234-237)— **关键指标 6**
- 预期:`./chimera-windows-x86_64.exe --version` 输出匹配 `^(aether|chimera) [0-9]+\.[0-9]+\.[0-9]+`(如 `chimera 1.5.1` 或 `aether 1.5.1`)
- binary 内部 codename 为 `aether`(`crates/chimera-cli/Cargo.toml [[bin]]`),CI 重命名为 `chimera`;二者均符合正则
- 失败处理:见 §7.2 指标 6

**Step 12: Verify binary size < 50MB**(L239-251)— **关键指标 7**
- 预期:日志出现 `OK: Binary size <N> bytes < 50MB limit`,其中 N < 52428800(50×1024×1024)
- 典型值:release profile 配置 `lto=true` + `opt-level="z"` + `strip=true`,典型 binary 7-15MB;Windows GNU binary 通常比 Linux 略大(含 mingw 运行时),但仍应 < 20MB
- 失败处理:见 §7.2 指标 7

**Step 13: Upload artifact**(L253-258)— **关键指标 8**
- 预期:退出码 0;日志出现 `Artifact chimera-windows-x86_64.exe was successfully uploaded`;workflow run 详情页 Artifacts 区可见 `chimera-windows-x86_64.exe`

## 6. 失败处理方案

### 6.1 通用流程

1. **下载日志**:在失败 job 页面点击右上角 `...` → `Download logs`,保存完整日志便于离线分析
2. **下载 build.log**(仅 build job 失败时):在 workflow run 详情页 Artifacts 区下载 `build-log-<target>`,内含完整 `cargo build` 输出
3. **定位失败 step**:按 §4 顺序对照日志,确认首个失败的 step
4. **执行对应诊断**:参见 §6.2-6.5
5. **修复后重测**:修复后删除已推送的 tag,重新打 tag 推送触发 CI
   ```bash
   # 删除远程 tag
   git push origin :refs/tags/v1.5.1-omega
   # 删除本地 tag
   git tag -d v1.5.1-omega
   # 修复后重新打 tag(在新 commit 上)
   git tag v1.5.1-omega
   git push origin v1.5.1-omega
   ```
6. **记录教训**:将根因与修复方案填写到 §8 验证报告模板,必要时同步到 `project_memory.md`

### 6.2 Windows GNU MinGW 失败场景(P1 重点)

> 完整的深度根因分析见姐妹文档 [`v1.5.1-omega_ci_windows_gnu_verification.md`](./v1.5.1-omega_ci_windows_gnu_verification.md) §4.2-4.8。本节给出快速诊断索引。

| 指标 | 失败现象 | 根因 | 修复方向 |
|------|---------|------|---------|
| 1 | `msys2/setup-msys2@v2` 安装失败 | 网络瞬时失败 / action upstream 回归 / MSYS2 镜像问题 | 重试 / 锁定 commit SHA / `update: false` |
| 2 | `Configure MinGW linker` 写入失败 | `GITHUB_ENV`/`GITHUB_PATH` 写入失败(极少见) / `if` 条件不匹配 | 重试 / 检查 matrix.target 拼写 |
| 3 | `which gcc` 返回空或非 `C:/msys64/...` 路径 | `GITHUB_PATH` 注入未生效 / PATH 优先级被覆盖 | 显式 `export PATH` / 排查重复 PATH 注入 |
| 4 | `gcc --version` 报错 | MSYS2 安装不完整 | `setup-msys2` 增加 `update: true` |
| 5 | `Build binary` 失败 | linker 问题 / ar 缺失 / 运行时混用 / OOM / 纯编译错误 | 见 §6.2.1 |
| 6 | `--version` 输出异常 | binary panic / 版本号不匹配 / 缺失子命令 | 下载 artifact 本地 `RUST_BACKTRACE=1` 运行 / 同步版本号 |
| 7 | Binary 体积超 50MB | 依赖膨胀 / profile 失效 | 评估新依赖 / 恢复 release profile |
| 8 | Artifact 上传失败 | 名称冲突 / 网络 / retention 配置 | 修正 artifact_name / 重试 |

#### 6.2.1 Build binary 失败的错误类型区分

| 错误信息 | 类型 | 修复方向 |
|---------|------|---------|
| `linking with cc failed` / `cannot find gcc` | linker 路径问题 | 回看指标 3-4,修复 MinGW 配置 |
| `cannot find ar` / `windres` | PATH 未注入 | 回看指标 2-3,确认 `GITHUB_PATH` 注入 |
| `undefined reference to '_imp__...'` | MinGW 与 MSVC 运行时混用 | 检查 `Cargo.toml` 是否引入 `windows-msvc` 专属依赖,改为 target-specific |
| `error[E0xxx]` | 纯 Rust 编译错误 | 与 MinGW 无关,本地应已捕获,本地修复后重新打 tag |
| `STATUS_STACK_BUFFER_OVERRUN (0xC0000409)` | 实际为 OOM(`__fastfail` 的 `FAST_FAIL_FATAL_APP_EXIT`) | 在 `Build binary` 步骤添加 `CARGO_BUILD_JOBS=2` 环境变量(与本地 `--jobs 2` 一致) |

| `download-artifact` 失败 | artifact 过期或名称不匹配 | 检查 build/deb job 的 artifact_name 唯一性 |

### 6.6 tag 回滚流程

> 当验证严重失败需要回滚时:

```bash
# 1. 删除远程 tag(触发 release.yml 的 build/docker/deb/release job 取消,已完成的 Release 需手动删除)
git push origin :refs/tags/v1.5.1-omega

# 2. 删除本地 tag
git tag -d v1.5.1-omega

# 3. 删除 GitHub Release(若已创建)
# GitHub repo → Releases → v1.5.1-omega → Delete

# 4. 删除 GHCR 镜像(若已推送)
# GitHub repo → Packages → 对应 package → Settings → Delete this package

# 5. 修复后重新打 tag
git tag v1.5.1-omega
git push origin v1.5.1-omega
```

**注意**:
- 删除 tag 不会自动取消正在运行的 workflow run,需手动到 Actions tab 取消
- 删除 GitHub Release 不会删除已上传的 GHCR 镜像,需单独删除
- 回滚后应填写 §8 验证报告,记录失败原因
