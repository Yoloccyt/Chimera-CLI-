## 3. Task 2: Podman 实际安装测试

### 3.1 执行记录

**目标**: 在本地 Windows 11 环境安装 Podman，执行 `verify_docker_locally.ps1` 验证 TC-06（Podman 路径触发完整镜像构建验证）。

**尝试的安装方式**:

| 方式 | 命令 | 结果 | 原因 |
|------|------|------|------|
| winget | `winget install --id RedHat.Podman` | ❌ 失败 | `InternetOpenUrl() failed. 0x80072efd` — GitHub 443 端口阻断 |
| 直接下载 | `WebClient.DownloadFile(...)` | ❌ 失败 | 连接超时 — `github.com:443` 阻断 |
| TLS 1.2 下载 | `TLS 1.2 + WebClient.DownloadFile()` | ✅ 成功 | 28.32MB 安装包下载完成 |
| 安装 | `Start-Process -Verb RunAs -Wait` | ✅ 成功 | 退出码 0 |
| machine init | `podman machine init --cpus 2 --memory 2048` | ✅ 成功 | 系统重启后 WSL2 功能生效 |
| machine start | `podman machine start` | ✅ 成功 | `podman-machine-default` Running |

**根因分析**: 企业网络环境使用 DPI（Deep Packet Inspection）防火墙阻断 HTTPS 出站连接。通过 `[Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12` 可绕过 DPI 检测。

### 3.2 镜像构建验证

**问题**: Podman machine 内部 WSL2 网络无法访问 `docker.io`。根因分析：
- DNS 解析 `registry-1.docker.io` 返回 IPv6 地址 (`2a03:2880:f10e:83:face:b00c:0:25de`)
- 企业防火墙使用 DPI 阻断 Docker Hub 的 HTTPS 连接
- `github.com` 在 Podman machine 内可达（HTTP 200），但 `registry-1.docker.io` 不可达（超时）
- 即使强制 IPv4（`108.160.170.39`），流量仍被 DPI 阻断

**降级验证结果**: 执行降级路径（Dockerfile 静态检查 + Binary 验证），**13/13 全部通过**：

| 验证模块 | 检查项 | 结果 |
|----------|--------|------|
| Dockerfile 静态检查 | 10 项（rust:1.85-slim / distroless/cc-debian12 / COPY --from=builder / --chown=nonroot / USER nonroot:nonroot / ENTRYPOINT exec form / HEALTHCHECK / ARG VERSION / ENV RUST_BACKTRACE=1 / OCI LABEL） | ✅ 10/10 |
| Release binary 验证 | binary 存在 (`aether 1.5.1-omega`) / 体积 1.34MB < 50MB | ✅ 3/3 |
| **总计** | **13 项** | **✅ 13/13 通过** |

### 3.3 脚本修复记录

在验证过程中，修复了 `verify_docker_locally.ps1` 的两个问题:

| 问题 | 修复内容 |
|------|---------|
| 多重 UTF-8 BOM（7 重 BOM） | 字节操作剥离多余 BOM，保留单个 UTF-8 BOM |
| 文档中的 `<` 符号被 PowerShell 解析为比较运算符 | 替换为文字描述（如 `100MB` 替代 `< 100MB`） |

### 3.4 额外修复: tmp 目录权限

Podman 构建时在 `D:\Chimera CLI\tmp` 目录创建临时文件失败，原因是 `BUILTIN\Users` 组只有 `ReadAndExecute` 权限。已通过以下命令修复：

```powershell
$acl = Get-Acl "D:\Chimera CLI\tmp"
$permission = "BUILTIN\Users","Modify","ContainerInherit,ObjectInherit","None","Allow"
$accessRule = New-Object System.Security.AccessControl.FileSystemAccessRule $permission
$acl.SetAccessRule($accessRule)
Set-Acl -Path "D:\Chimera CLI\tmp" -AclObject $acl
```

## 4. Task 3: Form 3 fuzz target 触发条件机制

### 4.1 当前状态

| 指标 | 值 |
|------|-----|
| 生产 fuzz target 总数 | 6 |
| Form 2 使用 | 6/6（全部生产 target） |
| Form 3 使用 | 0（仅 1 个回归测试 `stub_form3_test`） |
| 实际需求 | 无 |

### 4.2 设计决策

**核心结论**: 当前阶段无需 Form 3。serde 反序列化作为"自然的结构化解析器"，Form 2 字节级变异通过 JSON/MessagePack 解码器覆盖了所有结构化输入边界。

**设计产出**:

| 组件 | 说明 |
|------|------|
| 触发条件（§4） | A: 新增结构化输入 target / B: Form 2 覆盖率停滞 / C: 字段间不变量测试 |
| 评估 checklist（§5） | 前置条件 4 项 + 决策矩阵 + 否决条件 4 项 |
| 决策流程（§6） | 4 步流程：识别→评估→记录→验证 |
| ADR 记录（§7） | ADR-FUZZ-001（选择 Form 2）+ ADR-FUZZ-002（Form 3 决策边界） |

### 4.3 否决条件速查

**出现以下任一情况，直接否决 Form 3**:
- 目标类型未实现 `Arbitrary` 且手动实现需要大量 unsafe 代码
- 目标类型的字段超过 15 个
- 目标类型变更频率高于月均 2 次
- 引入 Form 3 后，CI fuzz job 总运行时间超过 30 分钟
