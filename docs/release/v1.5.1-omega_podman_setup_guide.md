## 1. Podman 简介与优势

Podman(Pod Manager)是 Red Hat 主导开发的开源容器引擎,遵循 OCI(Open Container Initiative)规范,设计目标是提供与 Docker CLI 兼容的使用体验,同时消除 Docker Desktop 在企业环境下的许可证与单点故障问题。截至 2026 年 7 月,Podman 主线版本为 5.x 系列(Podman Desktop 1.27.x),Podman 6.0 也已发布,本指南适用于 5.x 与 6.x 两个大版本。

### 1.1 与 Docker Desktop 的对比

| 特性 | Docker Desktop | Podman |
|------|----------------|--------|
| 许可证 | 商业(>250 员工或 >$10M 年收入企业需付费订阅) | 开源(Apache 2.0,完全免费) |
| 架构 | 守护进程(daemon-based,`dockerd` 常驻) | 无守护进程(daemonless,fork-exec) |
| 权限模型 | daemon 默认以 root 运行 | 原生 rootless(基于 user namespaces) |
| CLI 兼容性 | 原生 Docker CLI | Docker CLI 兼容(drop-in 替代) |
| 镜像格式 | OCI / Docker | OCI 优先(完全兼容 Docker 镜像) |
| Windows 后端 | WSL2 / Hyper-V | WSL2 / Hyper-V |
| GUI 工具 | Docker Desktop(闭源) | Podman Desktop(开源,可选) |
| 多阶段构建 | 支持 | 支持(底层调 Buildah) |
| distroless 镜像 | 支持 | 完全支持(OCI 标准) |
| Docker API 兼容 | 原生 | 提供 `podman system service` 兼容端点 |
| Kubernetes Pod | 不支持(需 Compose) | 原生支持(`podman pod`) |
| 单点故障 | daemon 崩溃影响所有容器 | 无 daemon,容器相互独立 |

### 1.2 选择 Podman 的核心理由

针对 Chimera CLI 项目特点,选择 Podman 的关键理由如下:

1. **无许可证限制** — 企业环境无需购买 Docker Desktop 商业许可,契合 Chimera CLI 开源(Apache-2.0)定位与企业防火墙/许可证受限场景
2. **无守护进程** — 每个 `podman` 命令独立 fork-exec,无 `dockerd` 常驻进程,降低攻击面与资源占用,契合项目 `#![forbid(unsafe_code)]` 安全哲学
3. **原生 rootless** — 容器以普通用户身份运行,契合项目"最小权限"安全哲学(`Dockerfile` 中 `USER nonroot:nonroot` 的同源理念)
4. **Docker CLI 兼容** — `scripts/verify_docker_locally.ps1` 中 `Invoke-EngineVerification` 函数对 `docker` 与 `podman` 使用完全相同的命令序列(`build` / `run --version` / `image inspect --format`),无需为 Podman 单独适配
5. **OCI 规范合规** — 与项目 `Dockerfile` 中的 OCI 标准 LABELS(`org.opencontainers.image.title` / `source` / `licenses` / `version`)理念完全一致
6. **distroless 镜像完美兼容** — Podman 完全支持 OCI 标准镜像,`gcr.io/distroless/cc-debian12` 无需任何特殊配置

### 1.3 架构说明

Podman 在 Windows 上的工作原理:

```
Windows PowerShell / CMD
    │
    │  podman 命令(本地 CLI)
    ▼
podman.exe(Windows 原生二进制)
    │
    │  远程通信(SSH over WSL2 网络)
    ▼
Podman Machine(WSL2 轻量 Linux VM,基于 Fedora CoreOS)
    │
    │  容器运行时(OCI runtime)
    ▼
crun / runc → Linux 容器(运行 chimera binary)
```

**关键点**:Windows 上的 Podman 通过 WSL2 提供的 Linux 内核运行容器。Podman machine 本质上是一个由 WSL2 驱动的轻量级 Linux 虚拟机,内部预装了 Podman、conmon、slirp4netns / pasta 等全套 OCI 运行时栈。

> ⚠️ **限制说明**:Podman 只能运行 Linux 容器,无法运行 Windows 容器。Chimera CLI 的 `Dockerfile` 基于 `rust:1.82-slim`(构建阶段)和 `gcr.io/distroless/cc-debian12`(运行阶段),均为 Linux 镜像,完全兼容。

## 2. 可行性调研结论

在投入安装前,本节先回答四个关键可行性问题。

### 2.1 无许可证限制特性

**结论**:Podman 与 Podman Desktop 均以 Apache 2.0 许可证开源,无任何规模/收入/员工数门槛。

- Docker Desktop 自 2021 年起对 >250 员工或 >$10M 年收入的企业要求付费订阅,违规使用存在法律风险
- Podman 由 Red Hat 主导开发,代码托管在 `github.com/containers/podman`,接受社区贡献,无商业版/社区版区分
- Podman Desktop 同样开源(`github.com/podman-desktop/podman-desktop`),无 Docker Desktop 那样的"个人版/专业版/企业版"分层

### 2.2 Docker CLI 兼容性

**结论**:Podman CLI 是 Docker CLI 的近完全 drop-in 替代,`scripts/verify_docker_locally.ps1` 中使用的三个核心命令均完美兼容。

| Docker 命令 | Podman 等价命令 | 兼容性 |
|-------------|-----------------|--------|
| `docker build -t <name> <path>` | `podman build -t <name> <path>` | 完全兼容(底层调 Buildah) |
| `docker run --rm <image> --version` | `podman run --rm <image> --version` | 完全兼容 |
| `docker image inspect <image> --format '{{.Size}}'` | `podman image inspect <image> --format '{{.Size}}'` | 完全兼容(Go template 语法一致) |
| `docker rmi <image> --force` | `podman rmi <image> --force` | 完全兼容 |
| `docker ps` / `docker images` | `podman ps` / `podman images` | 完全兼容 |
| `docker pull` / `docker push` | `podman pull` / `podman push` | 完全兼容(OCI registry 协议) |

> ★ **Insight**:`verify_docker_locally.ps1` 的 `Invoke-EngineVerification` 函数刻意将引擎名作为参数 `$Engine` 传入,然后用 `& $Engine build ...` 动态调用。这种设计使得 Docker 与 Podman 完全等价处理,无需任何分支逻辑——这是 Podman CLI 兼容性的最佳证明。

### 2.3 Windows WSL2 环境支持

**结论**:Podman 在 Windows 上原生依赖 WSL2(或 Hyper-V),WSL2 是推荐且性能最优的方案。

- Windows 11 Build 22000+ / Windows 10 Build 19043+ 均支持
- Podman machine 默认基于 Fedora CoreOS 的精简镜像,通过 WSL2 提供完整 Linux 内核
- WSL2 提供原生虚拟化性能(接近裸机 I/O),远优于 VirtualBox 等传统 VM
- 项目开发环境已是 Windows 11,无需额外操作系统升级

### 2.4 与 distroless 基础镜像的兼容性

**结论**:Podman 完全支持 `gcr.io/distroless/cc-debian12` 镜像,无需任何特殊配置。

- distroless 镜像遵循 OCI 镜像规范,Podman 原生支持 OCI 镜像
- `cc-debian12` 包含 glibc 动态链接器,可运行 GNU 链接的 Rust binary(本项目 `aether.exe` 链接方式匹配)
- Dockerfile 中的 `ENTRYPOINT ["chimera"]`(exec form,无 shell)被 Podman 正确解析执行
- `USER nonroot:nonroot`(UID 65532)在 Podman 下行为与 Docker 一致
- `HEALTHCHECK` 指令 Podman 4.0+ 完全支持(本项目使用 `CMD ["chimera", "--version"]` exec form)

> ★ **Insight**:distroless 镜像无 shell,这要求容器引擎必须支持 exec form 的 `ENTRYPOINT` / `CMD` / `HEALTHCHECK`。Podman 完全遵循 OCI runtime spec,exec form 是其原生支持的格式,不存在 shell form 退化问题。

## 3. 前置条件检查

### 3.1 系统要求

| 项目 | 要求 | 说明 |
|------|------|------|
| 操作系统 | Windows 11 Build 22000+ 或 Windows 10 Build 19043+ | Windows 11 推荐 |
| 架构 | x86_64 | ARM64 也支持,但本指南未覆盖 |
| 内存 | 至少 6 GB(推荐 8 GB+) | 供 Podman machine VM 使用 |
| 磁盘空间 | 至少 20 GB | WSL2 虚拟硬盘 + 容器镜像层 |
| CPU 虚拟化 | BIOS/UEFI 启用 Intel VT-x 或 AMD-V | WSL2 必需 |
| Windows 功能 | WSL2 | Podman 在 Windows 上的唯一依赖 |

### 3.2 检查硬件虚拟化

打开 PowerShell,执行:

```powershell
# 方法 1:任务管理器(直观)
# 任务管理器 → 性能 → CPU → 查看"虚拟化"是否为"已启用"

# 方法 2:系统信息
systeminfo | Select-String "Hyper-V"

# 方法 3:WMI 查询 CPU 虚拟化固件支持
(Get-CimInstance -ClassName Win32_Processor).VirtualizationFirmwareEnabled
# 输出 True 表示 BIOS 中已启用
```

若输出为 `False`,需重启进入 BIOS/UEFI 启用虚拟化:

- **Dell**:F2 → Advanced → CPU Configuration → Virtualization Technology
- **Lenovo**:F1/F2 → Security → Virtualization → Intel VT-x
- **HP**:F10 → System Configuration → Device Configurations → Virtualization Technology
- **ASUS**:DEL → Advanced → CPU Configuration → SVM Mode(AMD)/ Intel Virtualization Tech

### 3.3 检查并启用 WSL2

```powershell
# 以管理员身份运行 PowerShell

# 1. 更新 WSL 内核到最新版本
wsl --update

# 2. 启用 WSL 功能但不安装默认 Ubuntu 发行版
#    (Podman machine 会创建自己的 Fedora 发行版,不需要 Ubuntu)
wsl --install --no-distribution

# 3. 重启计算机
Restart-Computer

# 4. 重启后验证 WSL2 状态
wsl --status
# 期望输出:
#   默认版本: 2
#   内核版本: 5.15.x.x-1 或更高
#   WSL 版本: 2.x.x.x

# 5. 设置 WSL2 为默认版本(若尚未设置)
wsl --set-default-version 2
```

### 3.4 启用 Windows 功能(WSL2 不可用时的备用方案)

如果 `wsl --install --no-distribution` 失败(常见于 Windows 10 LTSC 或企业定制版),手动启用所需 Windows 功能:

```powershell
# 以管理员身份运行 PowerShell
dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart

# 重启计算机
Restart-Computer
```

### 3.5 企业代理环境预检(受限网络场景)

若开发机处于企业防火墙后,需先确认代理可达:

```powershell
# 测试 GitHub Releases 可达性(Podman 安装包源)
Invoke-WebRequest -Uri "https://github.com/containers/podman/releases" -UseBasicParsing -Method Head |
    Select-Object StatusCode

# 测试 podman.io 可达性
Invoke-WebRequest -Uri "https://podman.io" -UseBasicParsing -Method Head |
    Select-Object StatusCode

# 测试 gcr.io 镜像仓库可达性(distroless 基础镜像源)
Invoke-WebRequest -Uri "https://gcr.io" -UseBasicParsing -Method Head |
    Select-Object StatusCode

# 若需配置代理,在 PowerShell 会话中设置(临时)
$env:HTTP_PROXY  = "http://<proxy-host>:<proxy-port>"
$env:HTTPS_PROXY = "http://<proxy-host>:<proxy-port>"
$env:NO_PROXY    = "localhost,127.0.0.1,.internal.company.com"
```

> 若上述任一不可达,优先使用 §4.3 的"离线安装包(restricted environment installer)"方案。

## 4. Windows 安装步骤

Chimera CLI 项目推荐三种安装方式,按优先级排列:**winget**(最简单)> **官方安装包**(受限环境)> **Chocolatey / Scoop**(已有包管理器时)。

### 4.1 方式一:winget 安装(推荐)

Windows Package Manager(winget)是 Windows 11 内置的包管理器,无需额外安装。

```powershell
# 1. 安装 Podman CLI(必装,提供 podman 命令)
winget install --id RedHat.Podman -e

# 2. 安装 Podman Desktop(可选,GUI 管理工具)
winget install --id RedHat.Podman-Desktop -e

# 3. 关闭并重新打开 PowerShell 终端,使 PATH 生效

# 4. 验证安装
podman --version
# 期望输出:podman version 5.x.x / 6.x.x

podman info
# 显示系统信息,确认安装完整
```

**参数说明**:
- `--id RedHat.Podman`:指定 Red Hat 官方包 ID,避免第三方重名包
- `-e`:精确匹配包 ID,避免歧义

### 4.2 方式二:官方安装包(适用于受限环境)

适用于企业防火墙阻止 winget 仓库访问,或需要离线安装的场景。

#### 4.2.1 下载安装包

```powershell
# 方法 A:浏览器下载(交互式)
Start-Process "https://podman.io/downloads"
# 在浏览器中找到 Windows installer,下载 .exe 文件

# 方法 B:PowerShell 下载(需联网,从 GitHub Releases 拉取)
$ProgressPreference = 'SilentlyContinue'
$podmanReleases = "https://api.github.com/repos/containers/podman/releases/latest"
$latestRelease  = Invoke-RestMethod -Uri $podmanReleases -Headers @{ "User-Agent" = "PowerShell" }
$windowsAsset   = $latestRelease.assets | Where-Object { $_.name -match "podman.*setup.*x64\.exe$" } | Select-Object -First 1
Invoke-WebRequest -Uri $windowsAsset.browser_download_url -OutFile "$env:TEMP\podman-setup.exe"

# 方法 C:受限环境使用"离线安装包"
# 访问 https://podman-desktop.io/downloads/windows
# 下载 "Installer for restricted environments"(airgap 版本)
# 该安装包含所有必需组件,无需联网下载额外资源
```

> ★ **Insight**:**离线安装包(airgap installer)** 是受限环境的关键利器。标准安装包在首次启动 Podman Machine 时会从网络拉取 Fedora CoreOS 镜像,而 airgap 安装包将所有资源(WSL 发行版镜像、Podman binary、依赖组件)打包在内,完全离线运行。这对企业内网无外网访问的开发机尤其重要。

#### 4.2.2 执行安装

```powershell
# GUI 安装(交互式)
Start-Process "$env:TEMP\podman-setup.exe"

# 静默安装(自动化场景,CI 友好)
Start-Process "$env:TEMP\podman-setup.exe" -ArgumentList "/S" -Wait
```

安装向导中,虚拟化提供程序选择 **WSLv2**(推荐,性能优于 Hyper-V 且无需 Pro/Enterprise 版本)。

### 4.3 方式三:Chocolatey / Scoop(已有包管理器时)

如果开发机已安装 Chocolatey 或 Scoop,可复用现有包管理器:

```powershell
# Chocolatey(管理员 PowerShell)
choco install podman-desktop
# Chocolatey 会自动安装 podman CLI 与 Podman Desktop

# Scoop(用户级,无需管理员)
scoop bucket add extras
scoop install podman-desktop
```

### 4.4 验证安装完整性

```powershell
# 1. 检查 podman CLI 可执行
Get-Command podman | Select-Object Name, Source, Version

# 2. 检查版本
podman --version
# 期望输出:podman version 5.x.x / 6.x.x

# 3. 检查配置文件位置
# Podman 配置默认位于:
#   - 用户级:%USERPROFILE%\.share\containers\storage.conf
#   - 系统级:C:\ProgramData\containers\storage.conf(若存在)
Test-Path "$env:USERPROFILE\.share\containers"

# 4. 检查 PATH 是否已更新
$env:PATH -split ';' | Where-Object { $_ -match 'podman' }
# 期望输出类似:C:\Program Files\RedHat\Podman
```

## 5. 初始化与配置 Podman Machine

Podman 在 Windows 上通过 "Podman Machine"(WSL2 Linux VM)运行容器,首次使用前必须初始化。

### 5.1 创建默认 Podman Machine

```powershell
# 1. 初始化默认 machine(默认配置:2 CPU / 2048 MB RAM / 20 GB 磁盘)
podman machine init

# 2. 启动 machine
podman machine start

# 3. 验证 machine 运行状态
podman machine list
# 期望输出:
#   NAME       VM TYPE   CREATED   LAST UP    CPUS  MEMORY    DISK SIZE
#   podman*    wsl       ...       Currently  2     2.1GB     21.0GB
```

### 5.2 自定义资源配置(推荐)

默认 2 GB 内存对构建 Rust 项目不够(Chimera CLI workspace 有 35 crate),建议创建时指定更多资源:

```powershell
# 删除默认 machine(若已存在)
podman machine stop
podman machine rm -f

# 用更大资源重新初始化(推荐 4 CPU / 8 GB RAM / 60 GB 磁盘)
podman machine init --cpus 4 --memory 8192 --disk-size 60

# 启动
podman machine start
```

**参数说明**:
- `--cpus 4`:分配 4 个虚拟 CPU(加速 `cargo build`)
- `--memory 8192`:分配 8 GB 内存(满足 Rust release 构建峰值需求)
- `--disk-size 60`:60 GB 磁盘(容纳 WSL2 虚拟硬盘 + 容器镜像层 + 构建缓存)

### 5.3 网络代理配置(受限环境)

若开发机在企业代理后,需配置 Podman Machine 使用代理:

#### 5.3.1 创建时启用 User Mode Networking

```powershell
# 创建 machine 时启用 user mode networking(绕过 VPN 路由问题)
podman machine init --user-mode-networking
podman machine start
```

**User Mode Networking** 的作用:将容器网络流量通过宿主机用户态进程转发,而非走 WSL2 默认的 NAT 桥接。这能解决 VPN 环境下容器内访问宿主机服务的可达性问题。

#### 5.3.2 配置 Podman Machine 内部代理

```powershell
# 进入 Podman Machine shell
podman machine ssh

# 在 machine 内设置代理环境变量
sudo tee /etc/environment << 'EOF'
HTTP_PROXY=http://<proxy-host>:<proxy-port>
HTTPS_PROXY=http://<proxy-host>:<proxy-port>
NO_PROXY=localhost,127.0.0.1,.internal.company.com
EOF

# 退出 machine
exit

# 重启 machine 使代理生效
podman machine stop
podman machine start
```

### 5.4 镜像存储位置调整

默认情况下,Podman Machine 镜像存储在 WSL2 虚拟硬盘中,位于:
```
%LOCALAPPDATA%\Podman\podman-machine-root-fs.tar
%LOCALAPPDATA%\Podman\podman-machine-default-disk.qcow2
```

如果 C 盘空间不足,可迁移到 D 盘:

```powershell
# 1. 停止并删除现有 machine
podman machine stop
podman machine rm -f

# 2. 设置 PODMAN_STORAGE 环境变量指向新位置(用户环境变量)
[Environment]::SetEnvironmentVariable("PODMAN_HOME", "D:\Podman", "User")

# 3. 创建目标目录
New-Item -ItemType Directory -Path "D:\Podman" -Force

# 4. 重新初始化 machine
podman machine init --cpus 4 --memory 8192 --disk-size 60
podman machine start

# 5. 验证存储位置
podman info | Select-String "GraphRoot"
# 期望输出指向 D:\Podman 下的路径
```

> ⚠️ **D 盘空间注意**:Chimera CLI 项目规则 `.trae/rules/nuxus规则.md` §10.5 已记录"D 盘空间管理"短板(回收站黑洞/应用商店缓存),迁移 Podman 存储后需定期清理 `docker system prune -a` 释放无用镜像层。

## 6. Docker CLI 兼容性配置

虽然 Podman CLI 与 Docker CLI 高度兼容,但某些第三方工具(如 `docker compose`、IDE 插件)硬编码调用 `docker` 命令。可通过以下方式实现完全兼容。

### 6.1 设置 docker 命令别名(PowerShell Profile)

```powershell
# 1. 打开 PowerShell Profile 文件(若不存在则创建)
if (-not (Test-Path $PROFILE)) {
    New-Item -ItemType File -Path $PROFILE -Force
}
notepad $PROFILE

# 2. 在 Profile 中添加别名(将 docker 命令重定向到 podman)
#    添加以下行到 Profile 文件末尾:
Set-Alias -Name docker -Value podman

# 3. 重新加载 Profile
. $PROFILE

# 4. 验证别名生效
docker --version
# 期望输出:podman version 5.x.x / 6.x.x(因为 docker 现在指向 podman)
```

> ★ **Insight**:`Set-Alias docker podman` 是最简单的兼容方案。所有调用 `docker build` / `docker run` / `docker ps` 的脚本与 IDE 插件都会透明地走到 `podman`,无需修改任何代码。但注意 `docker compose` 子命令 Podman 4.x 之前不支持(需用 `podman-compose`),Podman 5.x+ 已原生支持 `podman compose`。

### 6.2 启用 Docker API 兼容端点

某些工具(如 Testcontainers、Docker SDK for Python/Go)通过 Docker API socket 通信,而非 CLI。Podman 提供兼容端点:

```powershell
# 启动 Podman system service(Docker API 兼容模式)
# 该命令在后台运行,提供 npipe:////./pipe/docker_engine 端点

# 验证 Docker API 兼容端点可用
# 方法:用 curl 访问 Docker API(需 WSL 或 Git Bash)
```

对于 Windows 原生 Docker API(npipe)兼容:

```powershell
# 启动 Windows 命名管道端点(需要管理员权限)
# 该端点模拟 \\.\pipe\docker_engine,Docker SDK 可直接连接
```

### 6.3 配置容器 Registry 短名称解析

Podman 默认对短镜像名(如 `alpine`)会提示选择 registry,而 Docker 默认从 `docker.io` 拉取。为保持与 Docker 行为一致:

```powershell

$configDir = "$env:USERPROFILE\.config\containers"
New-Item -ItemType Directory -Path $configDir -Force

@'
unqualified-search-registries = ["docker.io", "quay.io"]

[[registry]]
location = "docker.io"
[[registry.mirror]]
location = "mirror.gcr.io"
'@ | Out-File -FilePath "$configDir\registries.conf" -Encoding utf8

# 验证配置
podman info | Select-String "registries"
```

## 8. 与 Chimera CLI 项目集成

### 8.1 集成原理:`verify_docker_locally.ps1` 自动检测 Podman

Chimera CLI 项目的 `scripts/verify_docker_locally.ps1` 脚本已内置 Podman 自动检测逻辑,采用三级降级策略:

```
路径 1: 检测到 docker 命令 → 使用 docker 完整镜像构建验证
路径 2: 检测到 podman 命令(无 docker)→ 使用 podman 完整镜像构建验证
路径 3: 两者均无 → 降级验证(Dockerfile 静态检查 + binary 体积 + CI 引导)
```

**核心检测代码**(摘自 `verify_docker_locally.ps1` 第 285-309 行):

```powershell
$hasDocker = Test-CommandAvailable "docker"
$hasPodman = Test-CommandAvailable "podman"

if ($hasDocker) {
    # 路径 1: Docker 完整验证
    $ok = Invoke-EngineVerification -Engine "docker"
} elseif ($hasPodman) {
    # 路径 2: Podman 完整验证 (Docker 不可用时的替代引擎)
    Write-Host "未检测到 Docker, 发现 Podman, 将使用 Podman 执行镜像构建验证。" -ForegroundColor Yellow
    $ok = Invoke-EngineVerification -Engine "podman"
} else {
    # 路径 3: 降级验证
    Write-Host "未检测到 Docker / Podman, 执行降级验证 ..." -ForegroundColor Yellow
    Invoke-DockerfileStaticCheck
    Invoke-BinaryVerification
    Show-CIQueryGuidance
}
```

`Invoke-EngineVerification` 函数对 docker 与 podman 使用完全相同的命令序列,无需任何分支:

```powershell
# 1. 构建镜像
& $Engine build -t $ImageName $RepoRoot

# 2. 验证 --version 输出格式(与 release.yml grep 一致)
$versionOutput = (& $Engine run --rm $ImageName --version 2>&1).ToString().Trim()
$versionMatch = $versionOutput -cmatch '^(aether|chimera) \d+\.\d+\.\d+'

# 3. 验证镜像体积 < 100MB
$sizeRaw = (& $Engine image inspect $ImageName --format '{{.Size}}' 2>&1).ToString().Trim()
$imageSize = [int64]$sizeRaw
$sizeOk = $imageSize -lt 104857600   # 100MB

# 4. 清理临时镜像
& $Engine rmi $ImageName --force
```

### 8.5 与 CI(`release.yml` docker job)的对齐

本地 Podman 验证与 CI 的 Docker 验证在以下维度完全等价:

| 验证项 | CI(release.yml) | 本地(Podman via verify_docker_locally.ps1) |
|--------|------------------|---------------------------------------------|
| 镜像构建 | `docker build -t <image> .` | `podman build -t chimera-cli:local-verify .` |
| --version 校验 | `docker run --rm <image> --version` + `grep -qE` | `podman run --rm <image> --version` + `-cmatch` |
| 体积断言 | `docker image inspect --format {{.Size}}` + `< 100MB` | `podman image inspect --format {{.Size}}` + `< 100MB` |
| 安全配置 | Dockerfile 静态检查(基础镜像/USER/ENTRYPOINT) | 同(脚本路径 3a 与路径 2 互不冲突) |

> 本地验证通过后,推送 `v1.5.1-omega` tag,CI `release.yml` docker job 在 Linux runner 上用 Docker 重新验证,两套环境的 OCI 镜像一致性由 Podman 与 Docker 共同遵循 OCI 规范保证。

## 9. 常见问题排查

### 9.1 WSL2 相关问题

#### 9.1.1 错误:WSL2 未安装或版本过低

**症状**:`podman machine init` 失败,提示 "WSL2 is not installed" 或 "WSL2 distribution could not be created"。

**排查步骤**:

```powershell
# 1. 检查 WSL2 状态
wsl --status
# 若输出 "找不到命令" 或默认版本不是 2,需重新安装 WSL2

# 2. 升级 WSL 内核
wsl --update

# 3. 设置默认版本为 2
wsl --set-default-version 2

# 4. 列出现有发行版
wsl --list --verbose
# 若存在 podman-machine-default 但 STATE 为 Stopped,手动启动:
wsl -d podman-machine-default
```

#### 9.1.2 错误:0x80370102 虚拟机无法启动

**症状**:`podman machine start` 失败,错误码 `0x80370102`,提示 "The virtual machine could not be started because a required feature is not installed."

**根因**:BIOS/UEFI 中未启用硬件虚拟化,或 Hyper-V 平台功能未启用。

**解决方案**:

```powershell
# 1. 检查 BIOS 虚拟化是否启用
(Get-CimInstance -ClassName Win32_Processor).VirtualizationFirmwareEnabled
# 若为 False,重启进入 BIOS 启用 Intel VT-x / AMD-V

# 2. 启用 Hyper-V 平台功能(即使使用 WSL2 也需要)
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart

# 3. 启用 Windows 虚拟机监控程序平台
dism.exe /online /enable-feature /featurename:HypervisorPlatform /all /norestart

# 4. 重启计算机
Restart-Computer
```

#### 9.1.3 错误:WSL2 内存占用过高

**症状**:Podman Machine 运行时,`vmmem` 进程占用大量内存(>4 GB)。

**解决方案**:配置 WSL2 全局内存限制:

```powershell
# 创建或编辑 %USERPROFILE%\.wslconfig
$wslConfigPath = "$env:USERPROFILE\.wslconfig"
@'
[wsl2]
memory=8GB
processors=4
swap=2GB
localhostForwarding=true
'@ | Out-File -FilePath $wslConfigPath -Encoding utf8

# 重启 WSL2 使配置生效
wsl --shutdown
podman machine start
```

### 9.2 网络代理问题

#### 9.2.1 错误:容器内无法访问外部网络

**症状**:`podman build` 时拉取 `rust:1.82-slim` 失败,提示网络超时或 connection refused。

**排查步骤**:

```powershell
# 1. 检查 Podman Machine 内部网络
podman machine ssh
# 在 machine 内执行:
ping -c 3 8.8.8.8
curl -v https://registry-1.docker.io/v2/
exit

# 2. 若 ping 失败,配置 machine 代理
podman machine ssh
sudo tee /etc/environment << 'EOF'
HTTP_PROXY=http://<proxy-host>:<proxy-port>
HTTPS_PROXY=http://<proxy-host>:<proxy-port>
NO_PROXY=localhost,127.0.0.1
EOF
exit

# 3. 重启 machine
podman machine stop
podman machine start

# 4. 重新构建镜像
podman build -t chimera-cli:local-verify .
```

#### 9.2.2 错误:VPN 环境下容器访问宿主机服务失败

**症状**:VPN 连接时,从容器内无法访问宿主机上暴露的端口(如 `host.docker.internal:8080`)。

**解决方案**:使用 User Mode Networking 重建 machine:

```powershell
podman machine stop
podman machine rm -f

# 用 user mode networking 重建
podman machine init --user-mode-networking --cpus 4 --memory 8192 --disk-size 60
podman machine start

# 验证:user mode networking 下,host.containers.internal 可达
podman run --rm alpine:latest ping -c 3 host.containers.internal
```

### 9.3 镜像存储问题

#### 9.3.1 错误:磁盘空间不足

**症状**:`podman build` 失败,提示 "no space left on device"。

**排查与清理**:

```powershell
# 1. 查看 Podman 磁盘使用情况
podman system df
# 输出示例:
#   TYPE           TOTAL   ACTIVE  SIZE        RECLAIMABLE
#   Images         12      3       5.2GB       4.1GB (78%)
#   Containers     5       0       256MB       256MB (100%)
#   Local Volumes  2       1       102MB       50MB (49%)

# 2. 清理未使用的镜像、容器、卷
podman system prune -a --volumes
# 该命令删除:
#   - 所有停止的容器
#   - 所有未被容器使用的网络
#   - 所有 dangling 镜像(<none>:<none>)
#   - 所有未被容器使用的镜像
#   - 所有未被容器使用的卷

# 3. 查看具体镜像列表,手动删除大镜像
podman images
podman rmi <image-id> --force

# 4. 清理构建缓存(Buildah 缓存)
podman build --no-cache -t chimera-cli:local-verify .
```

#### 9.3.2 错误:镜像存储位置不在预期盘符

**症状**:Podman 镜像存储在 C 盘,导致系统盘空间不足。

**解决方案**:见 §5.4 镜像存储位置调整,通过 `PODMAN_HOME` 环境变量迁移到 D 盘。

### 9.4 构建相关问题

#### 9.4.1 错误:cargo build 在容器内失败

**症状**:`podman build` 时 `RUN cargo build --release -p chimera-cli` 步骤失败。

**根因分析**:

1. **网络问题**:容器内无法访问 crates.io
2. **内存不足**:Rust release 构建需要 4+ GB 内存,默认 machine 2 GB 不够
3. **路径问题**:Windows 路径与 Linux 容器路径转换错误

**解决方案**:

```powershell
# 1. 检查 machine 内存配置(应 >= 4 GB)
podman machine inspect | ConvertFrom-Json | Select-Object -ExpandProperty Resources | Select-Object Memory

# 2. 若内存不足,重建 machine
podman machine stop
podman machine rm -f
podman machine init --cpus 4 --memory 8192 --disk-size 60
podman machine start

# 3. 网络问题:配置代理(见 §9.2.1)

# 4. 路径问题:确保在项目根目录执行 build,且 Dockerfile 路径正确
cd "D:\Chimera CLI"
podman build -t chimera-cli:debug -f Dockerfile .
```

#### 9.4.2 错误:distroless 镜像无法运行

**症状**:`podman run --rm chimera-cli:local-verify --version` 失败,提示 "no such file or directory" 或 "permission denied"。

**根因分析**:

1. **binary 链接方式不匹配**:Rust GNU 工具链链接的 binary 需要 glibc 动态链接器,`distroless/cc-debian12` 提供此链接器
2. **binary 架构不匹配**:在 Windows 上交叉编译 Linux binary 失败
3. **文件权限错误**:`COPY --chown=nonroot:nonroot` 未正确执行

**排查步骤**:

```powershell
# 1. 检查镜像内 binary 是否存在
podman run --rm chimera-cli:local-verify ls -la /usr/local/bin/chimera
# 期望输出:-rwxr-xr-x 1 nonroot nonroot ... /usr/local/bin/chimera

# 2. 检查 binary 架构
podman run --rm chimera-cli:local-verify file /usr/local/bin/chimera
# 期望输出:ELF 64-bit LSB shared object, x86-64, ...

# 3. 检查 dynamic linker
podman run --rm chimera-cli:local-verify ldd /usr/local/bin/chimera
# 期望输出:linux-vdso.so.1 / libgcc_s.so.1 / libc.so.6 / ld-linux-x86-64.so.2

# 4. 若 ldd 失败,说明 binary 与 distroless 镜像不兼容
#    确保 Dockerfile 中 builder 使用 rust:1.82-slim(Debian-based,与 cc-debian12 glibc 兼容)
```


#### 10.1.5 配置 Rootless 模式(推荐)

```bash
# 1. 安装 rootless 依赖
sudo apt install -y slirp4netns fuse-overlayfs uidmap   # Debian/Ubuntu
# 或
sudo dnf install -y slirp4netns fuse-overlayfs uidmap   # Fedora/RHEL

# 2. 配置 subordinate UID/GID 映射
grep $USER /etc/subuid
grep $USER /etc/subgid
# 若为空,添加映射:
sudo usermod --add-subuids 100000-165535 --add-subgids 100000-165535 $USER

# 3. 启用 lingering(用户注销后容器继续运行)
sudo loginctl enable-linger $USER

# 4. 测试 rootless
podman run --rm docker.io/library/alpine:latest echo "Rootless works"

# 5. 配置默认 registry
echo 'unqualified-search-registries = ["docker.io", "quay.io"]' |
    sudo tee /etc/containers/registries.conf.d/shortnames.conf
```

### 10.2 macOS 安装(需 Podman Machine)

macOS 上 Podman 通过虚拟机运行 Linux 容器,类似 Windows 但使用原生虚拟化(无需 WSL2)。

#### 10.2.1 安装 Podman CLI

```bash
# 方法 A:Homebrew(社区维护,非官方推荐但常用)
brew install podman

# 方法 B:官方安装包(推荐)
# 访问 https://podman.io/downloads
# 下载 macOS .pkg 安装包,双击安装
```

#### 10.2.2 初始化 Podman Machine

```bash
# 1. 创建默认 machine
podman machine init

# 2. 自定义资源(推荐)
podman machine init --cpus 4 --memory 8192 --disk-size 60

# 3. 启动 machine
podman machine start

# 4. 验证
podman info
```

#### 10.2.3 与 Docker Desktop for Mac 的兼容

```bash
# 设置 docker 别名(添加到 ~/.zshrc 或 ~/.bashrc)
echo 'alias docker=podman' >> ~/.zshrc
source ~/.zshrc

# 启动 Docker API 兼容端点
podman system service --time=0 &
```

### 10.3 跨平台通用验证

无论哪个平台,安装完成后均可执行以下验证(对齐 Chimera CLI 项目要求):

```bash
# 1. 版本检查
podman --version

# 2. 系统信息
podman info

# 3. 运行 hello-world
podman run --rm hello-world

# 4. 拉取项目使用的 distroless 基础镜像
podman pull gcr.io/distroless/cc-debian12

# 5. 进入 Chimera CLI 项目根目录,执行验证脚本(Linux/macOS 用 .sh 版本)
cd /path/to/Chimera\ CLI
./scripts/verify_docker_locally.sh
# 该脚本与 Windows 版逻辑一致,自动检测 docker → podman → 降级验证
```
