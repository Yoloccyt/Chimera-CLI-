# v1.5.1-omega CI Windows GNU MinGW 链接器验证清单

> 本清单用于在首次推送 `v*.*.*-omega` tag 触发 `release.yml` 时，系统性验证 Windows GNU target 的 MinGW 链接器配置是否在 CI 环境中真正生效。
>
> 对应规则：`.trae/rules/nuxus规则.md` §10.5 P1 短板 "release.yml Windows job 未安装 MinGW"。
>
> 本文件不修改 `release.yml`，仅作为验证操作手册与状态跟踪依据。

---

## 1. 背景

### 1.1 短板描述

`release.yml` 的 Windows job 使用 `windows-latest` runner 构建 `x86_64-pc-windows-gnu` target。该 runner 默认不安装 MinGW 工具链（无 `D:/msys64/...` 或 `C:/msys64/...`），而项目本地 `.cargo/config.toml` 硬编码指向 `D:/msys64/mingw64/bin/gcc.exe`（本地 D 盘工具链），导致 CI 上 GNU target 链接阶段可能因找不到 linker 而失败。

该短板在 §10.5 中标记为 `⚠️ 待验证/修复`，优先级 P1。

### 1.2 release.yml 已有的 MinGW 配置

`release.yml` L129-156 已通过以下三步覆盖本地 `.cargo/config.toml` 的 linker 路径：

| 步骤 | 行号 | 作用 |
|------|------|------|
| `Setup MSYS2 and MinGW (Windows GNU target)` | L129-135 | 使用 `msys2/setup-msys2@v2` 安装 MSYS2 + `gcc`（`msystem: MINGW64`） |
| `Configure MinGW linker for CI` | L137-147 | 设置 `CARGO_TARGET_X86_64_PC_WINDOWS_GNU_LINKER=C:/msys64/mingw64/bin/gcc.exe`，并将 `C:/msys64/mingw64/bin` 注入 `GITHUB_PATH` |
| `Verify MinGW toolchain (Windows)` | L149-156 | 执行 `which gcc` / `which ar` / `gcc --version` / `ar --version` 做软验证 |

### 1.3 本地与 CI 的路径差异

| 环境 | gcc 路径 | 来源 |
|------|---------|------|
| 本地（Windows 11 + D 盘工具链） | `D:/msys64/mingw64/bin/gcc.exe` | `.cargo/config.toml` `target.x86_64-pc-windows-gnu.linker` |
| CI（`windows-latest` runner） | `C:/msys64/mingw64/bin/gcc.exe` | `msys2/setup-msys2@v2` 默认安装路径 + `CARGO_TARGET_*_LINKER` 环境变量覆盖 |

**关键机制**：`CARGO_TARGET_<TRIPLE>_LINKER` 环境变量优先级高于 `.cargo/config.toml`，因此 CI 无需修改仓库文件即可覆盖 linker 路径。`GITHUB_PATH` 注入 `C:/msys64/mingw64/bin` 是为了让 cargo 能找到 `ar`、`windres` 等 MinGW 辅助工具（仅设置 linker 不足）。

### 1.4 为何需要验证

配置逻辑上正确，但 §10.5 标记为 "待验证/修复" 的原因：

1. **未在真实 tag 推送中跑通**：PR 触发的 `release.yml` 只运行 `test` job（`if: github.event_name == 'pull_request' || startsWith(github.ref, 'refs/tags/v')`，见 L269），`build` job 仅 tag 触发（L47 `if: startsWith(github.ref, 'refs/tags/v')`）。因此 Windows GNU 的 MinGW 配置从未在 CI 环境中实际验证。
2. **环境变量覆盖链路较长**：`msys2/setup-msys2@v2` → `GITHUB_ENV` → `GITHUB_PATH` → cargo 读取，任一环节失效都会导致链接失败。
3. **`windows-latest` 镜像可能变化**：GitHub-hosted runner 预装工具集会更新，可能影响 MSYS2 安装路径或 PATH 优先级。

## 3. 验证指标清单

进入 GitHub repo → Actions → 对应的 "Release" workflow run → 点击 `Build (x86_64-pc-windows-gnu)` job，按以下顺序逐项核验。

每项均需明确记录：**通过 / 失败 / 跳过**，并附日志关键行截图或文本片段。

### 指标 1：`msys2/setup-msys2@v2` action 执行成功

| 字段 | 内容 |
|------|------|
| 步骤名 | `Setup MSYS2 and MinGW (Windows GNU target)` |
| 通过条件 | 步骤退出码 0；日志中出现 `MSYS2 installed` / `Installing package gcc` 且无 error |
| 失败判定 | 步骤红色叉号；日志含 `Failed to install` / `network error` / `gcc package not found` |
| 备注 | 首次运行会下载 MSYS2 完整镜像（约 200MB），耗时 3-5 分钟属正常 |

### 指标 2：`Configure MinGW linker for CI` 步骤成功设置环境变量

| 字段 | 内容 |
|------|------|
| 步骤名 | `Configure MinGW linker for CI` |
| 通过条件 | 步骤退出码 0；`$GITHUB_ENV` 与 `$GITHUB_PATH` 写入成功（步骤日志无报错） |
| 失败判定 | 步骤红色叉号；或后续步骤环境变量未生效（需结合指标 3 判定） |
| 验证方式 | 本步骤日志较短，主要看退出码；真正生效需看指标 3 的 `which gcc` 输出路径 |

### 指标 3：`Verify MinGW toolchain (Windows)` 步骤输出正常

| 字段 | 内容 |
|------|------|
| 步骤名 | `Verify MinGW toolchain (Windows)` |
| 通过条件 | 同时满足以下全部：<br>1. `which gcc` 输出 `/c/msys64/mingw64/bin/gcc`（非 "gcc not in PATH"）<br>2. `which ar` 输出 `/c/msys64/mingw64/bin/ar`（非 "ar not in PATH"）<br>3. `gcc --version` 输出形如 `gcc.exe (Revxxxx, Built by MSYS2 project) 14.x.x`<br>4. `ar --version` 输出形如 `GNU ar 2.42` |
| 失败判定 | 任一 `which` 输出 "not in PATH"；或 `--version` 输出为空；或路径非 `C:/msys64/mingw64/bin/` 前缀 |
| 备注 | 若 `which gcc` 返回其他路径（如 `/usr/bin/gcc`），说明 PATH 优先级被覆盖，需排查 `GITHUB_PATH` 注入时机 |

### 指标 4：`Build binary` 步骤构建成功

| 字段 | 内容 |
|------|------|
| 步骤名 | `Build binary` |
| 通过条件 | 步骤退出码 0；日志末尾出现 `Finished \`release\` profile [optimized] target(s) in X.YZs`；无 `error: linking with cc failed` / `cannot find gcc` / `undefined reference` |
| 失败判定 | 步骤红色叉号；日志含 `error[E0xxx]` / `linking failed` / `ld: cannot find` / `gcc: error: No such file or directory` |
| 关键诊断区 | 日志中 `=== Rust toolchain info ===` 段确认 `stable-x86_64-pc-windows-gnu`；`=== MinGW toolchain info ===` 段确认 gcc/ar 路径；`=== Build start ===` 后为实际编译输出 |
| 备注 | 失败时本步骤会自动通过 `::error::` annotation 输出前 30 条 error，并在日志末尾打印 `tail -n 100 build.log`；同时触发 `Upload build log on failure` 步骤上传 `build.log` artifact（保留 7 天） |

### 指标 5：Binary 体积 < 50MB

| 字段 | 内容 |
|------|------|
| 步骤名 | `Verify binary size < 50MB` |
| 通过条件 | 日志出现 `OK: Binary size <N> bytes < 50MB limit`，其中 N < 52428800（50×1024×1024） |
| 失败判定 | 日志出现 `ERROR: Binary size <N> bytes exceeds 50MB (52428800 bytes) limit`；步骤退出码 1 |
| 典型值 | release profile 配置 `lto=true` + `opt-level="z"` + `strip=true`，典型 binary 7-15MB；若超 50MB 说明体积优化失效或引入重量级依赖 |
| 备注 | Windows GNU binary 通常比 Linux 略大（含 mingw 运行时），但仍应 < 20MB |

### 指标 6：Binary `--version` 输出匹配正则

| 字段 | 内容 |
|------|------|
| 步骤名 | `Verify binary runs` |
| 通过条件 | 日志中 `./chimera-windows-x86_64.exe --version` 输出匹配 `^(aether\|chimera) [0-9]+\.[0-9]+\.[0-9]+`（如 `chimera 1.5.1` 或 `aether 1.5.1`） |
| 失败判定 | 输出为空；或输出 "version check skipped"；或版本号不匹配 tag（如 tag 是 v1.5.1-omega 但 binary 输出 1.4.0） |
| 备注 | binary 内部 codename 为 `aether`（`crates/chimera-cli/Cargo.toml [[bin]]`），CI 重命名为 `chimera`；二者均符合正则 |

### 指标 7：Artifact 上传成功

| 字段 | 内容 |
|------|------|
| 步骤名 | `Upload artifact` |
| 通过条件 | 步骤退出码 0；日志出现 `Artifact chimera-windows-x86_64.exe was successfully uploaded`；workflow run 详情页 Artifacts 区可见 `chimera-windows-x86_64.exe` |
| 失败判定 | 步骤红色叉号；日志含 `upload failed` / `artifact name conflict` / `retention-days invalid` |

## 4. 失败处理流程

### 4.1 通用流程

1. **下载日志**：在失败 job 页面点击右上角 `...` → `Download logs`，保存完整日志便于离线分析
2. **下载 build.log**（仅指标 4 失败时）：在 workflow run 详情页 Artifacts 区下载 `build-log-x86_64-pc-windows-gnu`，内含完整 `cargo build` 输出（含警告与错误）
3. **定位失败指标**：按 §3 顺序对照日志，确认首个失败的指标编号
4. **执行对应诊断**：参见 §4.2-4.8
5. **修复后重测**：修复后删除已推送的 tag（`git tag -d v1.5.1-omega && git push origin :refs/tags/v1.5.1-omega`），重新打 tag 推送触发 CI
6. **记录教训**：将根因与修复方案追加到本文件 §6 "验证记录"，必要时同步到 `project_memory.md`

### 4.2 指标 1 失败（msys2/setup-msys2@v2 安装失败）

**诊断步骤**：
- 查看步骤日志中的 `Failed to install` 或 `network error` 行
- 确认 `msys2/setup-msys2` action 版本是否被锁定（`@v2` 是 release tag，可能因 upstream 发布新版引入回归）
- 检查 `windows-latest` runner 是否当日有公告（https://www.githubstatus.com/）

**修复方案**：
- 网络瞬时失败：重试（重新打 tag 推送）
- action 版本回归：临时锁定到具体 commit SHA，如 `msys2/setup-msys2@<sha>`
- MSYS2 镜像问题：在 `setup-msys2` 步骤添加 `update: false` 跳过首次更新（牺牲版本新鲜度换取稳定性）

### 4.3 指标 2 失败（环境变量设置失败）

**诊断步骤**：
- 确认步骤日志中 `echo "..." >> $GITHUB_ENV` 与 `echo "..." >> $GITHUB_PATH` 均执行
- 检查 `if: matrix.target == 'x86_64-pc-windows-gnu'` 条件是否生效（matrix.target 是否拼写正确）

**修复方案**：
- `GITHUB_ENV`/`GITHUB_PATH` 写入失败：极少见，通常是 runner 临时故障，重试
- 条件不匹配：检查 `release.yml` matrix 定义中 `target` 字段是否为 `x86_64-pc-windows-gnu`（连字符不能误写为下划线）

### 4.4 指标 3 失败（Verify MinGW toolchain 输出异常）

**诊断步骤**：
- `which gcc` 返回空或 "not in PATH"：说明 `GITHUB_PATH` 注入未生效或被后续步骤覆盖
- `which gcc` 返回非 `C:/msys64/mingw64/bin` 路径：说明 PATH 优先级问题（runner 预装的其他 gcc 抢先）
- `gcc --version` 报错：MSYS2 安装不完整或 gcc 包未正确安装

**修复方案**：
- PATH 未生效：在 `Verify MinGW toolchain` 步骤前显式 `export PATH="/c/msys64/mingw64/bin:$PATH"`（临时验证，长期需排查 `GITHUB_PATH` 时机）
- 路径冲突：在 `Configure MinGW linker for CI` 步骤改为 `echo "C:/msys64/mingw64/bin" >> $GITHUB_PATH`（已是这样，确认无重复 PATH 注入覆盖）
- gcc 损坏：在 `setup-msys2` 步骤增加 `update: true` 强制刷新 MSYS2 包数据库

### 4.5 指标 4 失败（Build binary 构建失败）

**诊断步骤**：
- 优先查看 `::error::` annotation（前 30 条 error 摘要）
- 下载 `build.log` artifact 查看完整错误
- 区分错误类型：
  - `linking with cc failed` / `cannot find gcc`：linker 路径问题（回看指标 3）
  - `cannot find ar` / `windres`：PATH 未注入 `C:/msys64/mingw64/bin`（回看指标 2-3）
  - `undefined reference to '_imp__...'`：MinGW 与 MSVC 运行时混用，检查是否有 crate 误链 MSVC 库
  - `error[E0xxx]`：纯 Rust 编译错误，与 MinGW 无关，本地应已捕获
  - `STATUS_STACK_BUFFER_OVERRUN (0xC0000409)`：clippy/build OOM，参考 §7.1 `--jobs 2` 缓解

**修复方案**：
- linker 问题：回到 §4.4 修复 MinGW 配置
- 运行时混用：检查 `Cargo.toml` 是否引入了 `windows-msvc` 专属依赖（如 `windows-sys` 某些 feature），改为 target-specific 依赖
- OOM：在 `Build binary` 步骤添加 `CARGO_BUILD_JOBS=2` 环境变量（与本地 `--jobs 2` 一致）
- 纯编译错误：本地修复后重新打 tag

### 4.6 指标 5 失败（Binary 体积超 50MB）

**诊断步骤**：
- 确认实际体积（日志中 `Binary size: <N> bytes`）
- 排查近期是否引入重量级依赖（`Cargo.lock` diff）
- 确认 release profile 配置未失效（`Cargo.toml` `[profile.release]` 仍为 `lto=true` + `opt-level="z"` + `strip=true`）

**修复方案**：
- 依赖膨胀：评估新依赖必要性，或拆分为 feature gate
- profile 失效：恢复 release profile 配置
- 临时放宽阈值：仅在确认体积增长合理（如新增核心功能）且经评审后，可临时调高 `release.yml` L247 的阈值（不推荐，应优先优化体积）

### 4.7 指标 6 失败（--version 输出异常）

**诊断步骤**：
- 输出为空：binary 可能 panic 在启动阶段，下载 artifact 本地运行 `./chimera-windows-x86_64.exe --version` 确认
- 版本号不匹配：`Cargo.toml` `workspace.package.version` 与 tag 不一致（§7.2 第 9 项检查遗漏）
- "version check skipped"：binary 退出码非 0，需查看 `./chimera-windows-x86_64.exe --version` 的 stderr 输出

**修复方案**：
- 版本号不匹配：更新 `Cargo.toml` `workspace.package.version` 为 tag 对应版本，重新打 tag
- binary panic：下载 artifact 本地用 `RUST_BACKTRACE=1` 运行定位 panic 位置
- 缺失子命令：检查 `chimera-cli` 的 clap 配置是否正确暴露 `--version`

### 4.8 指标 7 失败（Artifact 上传失败）

**诊断步骤**：
- `artifact name conflict`：matrix 中存在重名 artifact（检查 `artifact_name` 字段唯一性）
- `upload failed`：runner 与 GitHub artifact 存储之间的网络问题
- `retention-days invalid`：`retention-days: 30` 配置无效（应为 1-90 整数）

**修复方案**：
- 名称冲突：修正 matrix `artifact_name` 字段
- 网络问题：重试
- retention 配置：确认 `retention-days: 30` 在合法范围
